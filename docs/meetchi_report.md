## 第一段：前情提要

### 為什麼不繼續用 Meeting Ink？

2025 年公司原本評估導入 **Meeting Ink**（AI 會議記錄 SaaS），最終因兩個根本性問題而取消：

| 問題 | 說明 |
|------|------|
| **🔒 資料主權** | 所有會議記錄存放於 SaaS 服務商的 Azure 雲端（境外），機密會議（董事會、產品策略、客戶合約）外流風險無法消除，且合約上亦無法做到完整隔離 |
| **🔧 開發自由度** | SaaS 服務無法客製化模板、無法整合公司現有 ERP/CRM 流程，固定功能無法隨公司需求演進 |

**Meeting Ink 報價**：100 帳號 / 每月 1,000 小時 / 年訂閱 **NTD 140,000**

### 為什麼決定自開發？

> 「既然問題出在外部，解法就是把它拉進來」

取消 SaaS → 轉向自研，由內部工程師以開源技術自建，資料完全在公司掌控範圍內。

### MeetChi 命名由來

```
Meet（會議英文）+ Chi（奇美 CHIMEI 英文縮寫）
→ MeetChi

額外彩蛋：發音與麻將「聽牌」的日文 Migi 相近
設計理念：與會人只要開啟系統就好——聽就好，不需要做筆記
```

## 第三段：SaaS 服務可能無法負荷

### 量能缺口分析（第一性原理）

**Meeting Ink 方案**：100 帳號 / **每月 1,000 小時**上限

**實際需求**：每月 **2,200 小時**

```
缺口計算：
  每月需求 2,200hr - 方案包含 1,000hr = 超額 1,200hr/月

  超額時數費率：Meeting Ink 官方未公開，需另外購買
  若以行業均價 $0.5~1.0 USD/hr 估算：
  → 每月追加費用：1,200 × $0.5~1.0 × 31.7 ≈ NTD 19,000~38,000
  → 年度追加：NTD 228,000~456,000

  原方案 NTD 140,000 + 追加 NTD 228,000~456,000
  → 實際年費：NTD 368,000 ~ 596,000（約 37~60 萬/年）
```

### MECE 多維度無法負荷分析

| 面向 | Meeting Ink SaaS | MeetChi 自建 |
|------|-----------------|-------------|
| **量能（時數）** | ❌ 1,000hr/月 vs 實際 2,200hr/月，不足 55% | ✅ 無上限 |
| **帳號彈性** | ❌ 超過 100 帳號需付費（金額未公開） | ✅ 無限帳號 |
| **資料主權** | ❌ 資料在 Azure（境外雲端） | ✅ 資料在公司 GCP 新加坡區域 |
| **客製自由度** | ❌ 固定功能，無法改 | ✅ 完整控制摘要模板、流程 |
| **台灣繁中優化** | ⚠️ 英文優先，繁中/台語/中英混合品質有限 | ✅ Breeze-ASR-25 繁體中文最優化 |
| **機密會議保護** | ❌ 無法完全隔離，存在合規風險 | ✅ 可選地端部署，資料不外流 |

> **結論**：Meeting Ink 光是量能缺口就讓實際年費可能突破 60 萬，加上資料主權與功能限制，**SaaS 路徑的邊際成本遠高於自建**。

## 第四段：預期效益

### 核心效益：會議從「耗時行政」→「可搜尋知識資產」

> **白皮書第一性原理（MeetingInk，2025）**：
> 「會議紀錄的本質是人類協作數據的持久儲存機制。缺乏系統，這些決策就是企業最昂貴的隱形沉沒成本。」

**化工製程類比**：

> 「我們每天投入 100 場會議，但這些決策與共識大量流失在人工整理環節——就像製程裡沒有回收槽的中間產物。MeetChi 就是在這段管線裡加結構化的收集槽。」

**層次 1 — 時間節省（直接回收人力）**

| 指標 | 數值 |
|------|------|
| 每日需摘要的會議（SCR=40%） | 40 場 |
| 每場人工整理時間（現況） | 45~60 分鐘 |

**層次 2 — 決策品質提升（間接效益）**

## 第五段：計算方式

### ROI 計算（5 步驟）

```
Step 1 - 實際需摘要場次（基於 MeetingInk 白皮書 SCR 數據）：
  每日 100 場 × SCR 40% = 40 場（需結構化摘要）
  其餘 60 場：純轉錄查詢型

Step 2 - 每場節省時間：
  需摘要的 40場：人工 45~60 分 → AI 後 5~8 分 = 節省 37~55 分/場
  純轉錄的 60場：人工 15~20 分 → AI 後 2~3 分 = 節省 13~17 分/場

Step 3 - 每日總節省：
  40 × 37~55 + 60 × 13~17 = 2,260~3,220 分鐘/日 = 37.7~53.7 人時/日

Step 4 - 年節省金額（月薪 40K，時薪 250 元）：
  保守：100場 × 15分 × 251天 ÷ 60 × 250 = NTD 157 萬/年
  中性：251天 × 37.7hr × 250元 = NTD 236 萬/年

Step 5 - ROI：
  保守：(157萬 - 44.8萬) ÷ 44.8萬 = 250%
  中性：(236萬 - 44.8萬) ÷ 44.8萬 = 427%
```

**數據來源**：MeetingInk 白皮書 2025 Q2（30,000 場會議統計，SCR=40%）

### B.1 現有簡報結構評分

```
①前情提要 → ②情境假設 → ③SaaS無法負荷 → ④預期效益
→ ⑤計算方式 → ⑥方案比較 → ⑦目前進度 → ⑧未來方向
```

| 段落 | 優點 | 問題 |
|------|------|------|
| ① 前情提要 | 必要，建立正當性 | ✅ |
| ② 情境假設 | 誠實揭露假設 | ⚠️ 應先說 100 場的數字依據在哪（是全公司估算？還是未來目標？） |
| ③ SaaS 無法負荷 | 量能缺口有說服力 | ⚠️ 主管可能問：「為何不跟 Meeting Ink 重新議價？」需預備答案 |
| ④ 預期效益 | ROI 數字清晰 | ⚠️ 效益在進度之前，但系統還沒完成，主管可能問「這些效益是假設還是實測？」 |
| ⑤ 計算方式 | 透明，增加可信度 | ⚠️ 太詳細可能讓 5 分鐘簡報超時，可以備而不簡報 |
| ⑥ 方案比較 | 完整 | ✅ |
| ⑦ 目前進度 | 揭露現況 | ⚠️ 風險：主管可能反應「還沒完成就來報成本？」 |
| ⑧ 未來方向 | 展示發展性 | ✅ |

### B.2 盲點與建議

**盲點 1：「效益」在「進度」之前，但系統未完成** 🔴

- **風險**：主管聽到 ROI 250% 很好，然後聽到「摘要功能還在開發中」，心裡落差大
- **解法**：
  - (A) 調整順序：先說進度，再說效益（「以目前這個基礎，往上加建摘要後效益是...」）
  - (B) 或在效益前明確說：「**以下是完整導入後的預期效益**——目前已完成 70%」

**建議調整順序**：
```
前情提要 → 目前進度（先讓主管知道你有交付物）→ 情境假設
→ SaaS 無法負荷 → 效益（有了進度背書再說效益更可信）
→ 計算方式（備用，按需補充）→ 方案比較 → 未來方向
```

**盲點 2：SaaS 超額費用是你估的，不是官方數字** 🟡

- **風險**：主管追問「你怎麼知道超額費是多少？跟對方確認過嗎？」
- **解法**：明確標注「行業均價估算，待向 Meeting Ink 重新詢價確認」，主動提出「下一步行動」

**盲點 3：100 場/日是全公司假設，有沒有過度樂觀？** 🟡

- **風險**：主管問「100 場是哪來的，有統計過嗎？」
- **解法**：說明是情境假設用於成本計算，並提供**實際現況**（目前測試環境幾場？）作為起點。  
  或：先以「保守估算 20 場/日」作為導入初期基準，讓主管接受梯度導入的概念

**盲點 4：開發環境是個人 GCP，合規風險** 🟡

- **風險**：主管問「公司資料跑在員工個人帳號上？」
- **解法**：主動交代「測試資料僅使用 YouTube 公開影片，無任何公司機密資料進入個人 GCP 環境」

**盲點 5：無明確時程表** 🟠

- **風險**：主管問「MVP 什麼時候完成？搬遷什麼時候做？」
- **解法**：即使是模糊估計，也要給一個「預計 Q2 完成 MVP，Q3 提出環境申請」的時程

## 附錄 D：Plaud 中國背景資安風險分析（第一性原理 + MECE）

### D.1 第一性原理：資安的本質是「誰能存取你的資料」

| 層次 | 問題 | Plaud 狀況 |
|------|------|------------|
| **法律管轄權** | 企業實體在哪國法律下運作？ | ⛔ 深圳機智連接科技（iZYREC）→ 中國《國家安全法》管轄 |
| **技術存取權** | 誰可以改韌體/軟體/API？ | ⚠️ 硬體由深圳團隊製造，韌體更新權限歸屬不透明 |
| **資料流向** | 資料存在哪、經過哪、誰處理？ | ⚠️ 設備→App→AWS Oregon 雲端→ GPT/Claude/Gemini API |
| **合規認證** | ISO/SOC/GDPR 能保證什麼？ | ✅ 有 ISO 27001/SOC 2/HIPAA/GDPR，但認證只保證「流程合規」，不保證「法律要求時拒絕配合」 |

### D.2 MECE 風險維度

| 風險類別 | 嚴重度 | 說明 |
|---------|--------|------|
| **品牌身份不一致** | 🔴 高 | 行銷為「美國品牌 Plaud Inc.」，實際經營主體為深圳 iZYREC，創辦人許高（Xu Gao）主推「Day One Global」策略規避中國標籤 |
| **政治條款異常** | 🔴 高 | 美國官網服務條款曾出現「不得顛覆國家政權、破壞國家統一」等中國特有審查措辭 |
| **法律後門風險** | 🔴 高 | 中國《國家安全法》第七條：任何組織和公民都應當依法支持、協助和配合國家情報工作 |
| **資料上雲強制性** | 🟡 中 | 離線可錄音（本地儲存 64GB），但轉錄/摘要**必須**上傳雲端，敏感會議無法完全離線使用 |
| **AI 訓練資料風險** | 🟡 中 | Plaud 條款提及資料可能用於「服務改善」和「技術研發」，資料用途邊界模糊；反觀 Meeting Ink 明確承諾「絕不用於模型訓練」，MeetChi 自建則資料完全不經第三方 |
| **企業管控能力** | 🟡 中 | 無 IT Admin Console、無企業 SSO、無審計日誌——完全是消費級產品 |
| **供應鏈風險** | 🟡 中 | 若公司有「紅色供應鏈」排除條款，Plaud 可能直接被排除在採購清單外 |

### D.3 結論：Plaud 的定位與企業適用性

```
✅ 適用：個人使用者、Freelancer、小型團隊非機密用途
⛔ 不適用：涉及商業機密的企業會議、政府單位、有資安合規要求的組織

與 MeetChi 的根本差距：
  Plaud  → 消費級硬體 + 雲端 AI = 便利但不可控
  MeetChi → 企業級架構 + 可選地端 = 透明且完全可控
```
