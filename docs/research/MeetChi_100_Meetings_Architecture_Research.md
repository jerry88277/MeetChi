# 🚀 MeetChi 核心架構深度研究與戰略決策報告
**基準情境**：每日 100 場會議，均勻分佈於 `08:00 ~ 17:00` (共 9 小時)。
**分析方法論**：First Principles (第一性原理), MECE (不重疊不遺漏原則), SWOT 分析, Model-thinking。

---

## 子任務一：負載極限與物理限制推演 (第一性原理 & Model-thinking)

在討論模型前，我們必須先回歸「物理法則」與「基礎排隊理論」來建構流量模型 (Traffic Model)。

### 1-A. Concurrency (並發數) 模型推算
如果在 9 小時內要均勻消化 100 場會議，平均每小時約有 `11.1 場` 會議進行。
根據 Little's Law ($L = \lambda W$)，若假設每場會議平均時長為 1 小時：
* **常態並發量 (Average Concurrency)**：約 **11~12 路** 同時收音。
* **尖峰並發量 (Peak Concurrency)**：考慮 09:00 或 14:00 的會議群聚效應，系統必須容忍至少 **20 路並行 (20 Streams)**。

### 1-B. 算力與延遲的死亡交叉 (First Principles)
「流式語音辨識 (Streaming ASR)」本質是每隔 $N$ 秒送一段音軌。
如果使用單一處理緒 (Sequential worker)，單個 GPU 在沒有優化的情況下，處理 1 條短音軌需 0.12 秒 (RTF=0.12)。
當 **20 路併發** 同時湧入，若採用排隊機制處理，第 20 個人的語音會被延遲高達 $20 \times 0.12 = 2.4$ 秒。隨時間拉長，排隊延遲會無限堆疊（佇列溢出）。
>**結論**：要在這個情境存活，只有三條路：(1) 開更多機器分流、 (2) 強迫 GPU 在底層一次處理 20 個陣列 (Batching)、 (3) 請用戶自己的電腦算。這完美對齊了我們接下來要評估的三個架構。

## 子任務二：架構方案 A - 傳統擴展 `CT2 Faster-Whisper`

**核心機制**：將 CTranslate2 最佳化的 Faster-Whisper 部署在 Cloud Run (CPU/GPU) 上，由 Google Cloud Load Balancer (GLB) 負責橫向擴展 (Scale-out)。

### MECE 結構拆解
* **吞吐量 (Throughput)**：單機 4.4G VRAM 足跡穩定。據文獻指出，`CTranslate2` 將 GPU VRAM 消耗降低了 2~3 倍，並提升推理速度達 6~8 倍，極為適合「低延遲 (Low Latency)」需求。但在即時處理前提下，單一機器若承受超過 5 路同步，延遲將逐漸超過 1 秒。
* **資源成本 (Resource)**：為了支撐 20 路尖峰，系統必須在 Cloud Run 上長出 4 ~ 5 台 L4 GPU 容器。

### SWOT 分析
* **(S) 優勢：穩定性極致**。零幻覺迴圈，開箱即用，沒有冷啟動地獄。
* **(W) 劣勢：擴展成本**。每一路橫向擴展都是真金白銀的 L4 租金 ($約 0.7 USD/hour/台)，20 路並發等於每小時燒掉 3.5 美金。
* **(O) 機會：成熟的 Serverless**。隨插即用，雲端自動殺掉不用機器的機制最為純熟。
* **(T) 威脅：硬體利用率極低**。大部分時間 GPU 只在做零星秒數的運算，其餘時間都在閒置掛機，造成極大的運算力浪費。

## 子任務三：架構方案 B - `Whisper MLA + vLLM` (連續 Batching 兵器)

**核心機制**：單機部署 1 台大實例的 L4 GPU，強迫將 Cloud Run 並發度 (`--concurrency`) 調至 `80`。利用 vLLM PagedAttention 以及 MLA (潛伏注意力) 直接壓平 20 路的 KV Cache 膨脹。

### MECE 結構拆解
* **吞吐量 (Throughput) 與算力置換**：底層將 20 路的音訊 `[Batch=20]` 透過 Triton kernel 合併。**深度研究指出，MLA 本質上是「以算力換取記憶體」**：雖然 KV Cache 變小，但注意力機制在實時解壓縮 (Decompression) 時需要執行比傳統 MHA 多出約 4 倍的矩陣操作。因此高並發吞吐量的背後，極度仰賴卡片的原始算力。
* **記憶體 (VRAM KV 壓縮)**：這是最震驚的突破。根據最新文獻《Whisper-MLA》，引入 MLA 至 Whisper 的 Decoder 中，**精確減少了 87.5% 的 KV Cache 記憶體足跡**（效能僅犧牲 0.17% WER）。實測數據顯示，在極限情境 `[Batch=64, SeqLen=2048]` 之下，原版 Whisper 會直接衝破 24G 導致 OOM 崩潰，而 **Whisper-MLA 則穩穩停留在 15.4 GB**。這正是 L4 單卡扛下百場會議的物理基石。

### SWOT 分析
* **(S) 優勢：成本與吞吐量的雙重輾壓**。理論上「一台配置 L4 的容器」就能扛下 100 場會議的常態與尖峰並發。整體成本降為方案 A 的 1/5 甚至 1/10。
* **(W) 劣勢：開發難度地獄與功能丟失**。深度調查表明，雖然 vLLM 近期官方剛加入了對 Whisper (Encoder-Decoder 架構) 的支援，但**尚未支援精細的「字組級時間戳 (Word-level timestamps)」**；且針對我們客製化的 Breeze-ASR-MLA 架構，需重寫 Custom Registry 註冊。
* **(O) 機會：LLM 融合**。如果底層架好 vLLM，同一張卡甚至能在空閒時間載入文本模型做會議總結 (Co-location)。
* **(T) 威脅：單點失效風險 (SPOF) 與冷啟動**。百場會議賭在單點容器上，一旦發生系統 OOM 則全數斷線；此外 vLLM 冷啟動耗時長，會引發 504 Gateway Timeout。

## 子任務四：架構方案 C - `Edge-Cloud 混合架構 (Pure Web)`

**核心機制**：全面放棄伺服器端 ASR 算力。利用 `Transformers.js (WebGPU)` 或 `Whisper.cpp (Wasm)` 讓使用者的瀏覽器下載幾百 MB 的小型量化模型，在用戶端執行即時語音辨識，雲端僅做文字轉發與 LLM 分析。

### MECE 結構拆解
* **運算分布 (Compute Distribution)**：伺服器 ASR 負載率 $= 0\%$。客戶端負載率 $= 100\%$。
* **記憶體沙盒與洩漏致命傷 (Memory Leaks)**：根據前瞻研究與開源社群回報 (Issue #860)，使用 `Transformers.js (WebGPU)` 在瀏覽器跑長音頻時，若底層 Encoder KV Cache 的張量 (Tensor) 沒有被正確呼叫 `dispose()` 回收，會引發嚴重的**持續性記憶體洩漏**，最終造成瀏覽器崩潰或 `device is lost`。(需確保升級至 `>= 3.0.0-alpha.19` 修復版)。而 `Whisper.cpp (Wasm)` 則容易受限於嚴格的線性記憶體大小天花板 (Linear Memory Limits)。

### SWOT 分析
* **(S) 優勢：極致省錢與無限擴容**。100 場跟 10 萬場會議對後端 ASR 成本都是 $0。具備無懈可擊的資訊隱私安全 (聲音從未上傳)。
* **(W) 劣勢：硬體門檻與精度妥協**。必須仰賴 INT8/FP16 甚至 INT4 量化模型 (`tiny` 或 `base`) 來適應瀏覽器 Sandbox，辨識錯誤率 (WER) 大增。
* **(O) 機會：漸進式適配策略 (Adaptive Strategy)**。利用最新的 Web AI 檢測機制，在高階電腦上自動啟用 WebGPU 跑高品質模型，在舊手機則靜默降級至 Wasm + 迷你模型，確保不閃退。
* **(T) 威脅：不可控的客訴**。無法保證終端使用者的硬體設備水準。瀏覽器尚未啟用硬體加速等不可控環境將導致崩潰級的用戶體驗。

## 🛑 盲點與防禦性設計 (Blind Spot Analysis & Solutions)

我們刻意引入多種思維模型 (Model-thinking) 重新詰問上述結論，發現了最初分析在完美實驗室環境下，會忽略的「現實物理盲點」：

### 盲點一：排隊理論的「驚群效應 (Thundering Herd Problem)」
* **盲點描述**：一開始計算的「尖峰 20 路併發」假定流量也是連續分佈的。但現實世界中，**會議幾乎都在「整點」準時開始**（例如 09:00:00 湧入 30 場會議連線）。在這一瞬間，冷啟動的伺服器與資料庫連線池會瞬間耗盡，導致 503 服務不可用。
* **解決方案 (架構重塑)**：必須在 WebSocket 連入與 ASR 推論引擎間，插入 **Token Bucket限流器** 或是 **Pub/Sub 非同步緩衝隊列**。讓晚到的連線排隊或降級處理，並在前端實作 Jitter (隨機延遲重試)，打散瞬間波峰。

### 盲點二：資訊理論的「訊噪與幻覺放大 (Noise & Hallucination)」
* **盲點描述**：分析中假設「傳輸到後端的音檔都是完整的」。若使用 WebRTC/WebSocket 傳輸，在行動網路上必定會有 Packet Loss 或斷音。當 vLLM 或 CT2 收到「破碎的半截單字或雜音」時，Attention 機制會被干擾，導致模型產生瘋狂重複的幻覺 (例如：`......` 或 `謝謝觀看`)。
* **解決方案 (架構重塑)**：系統不能盲目依賴「定時切片 (例如每 1 秒切一包)」。必須在前端或雲端閘道器引入超輕量級的 **VAD (Voice Activity Detection)**。只在「確認有真實人聲」且「遇到語句停頓」時才打包送交 ASR，同時音檔切片需保持 `500ms` 的前後重疊 (Overlap)，從物理層面消滅幻覺的溫床。

### 盲點三：微觀經濟學的「Cloud Run GPU 隱藏待機稅 (Always-on Trap)」
* **盲點描述**：我們假設方案 B (vLLM 單卡) 最省錢。但 Cloud Run GPU 如果設置 `min-instances=1` 以避免 vLLM 長達一分鐘的恐怖冷啟動，代表這張 L4 即使在半夜完全沒有會議時，依然在默默計費。一個月將產生可觀的「待機稅」。
* **解決方案 (架構重塑)**：導入 **Wake-on-Demand** 設計。利用 Cloud Scheduler 或使用者登入系統的 API，在使用者「預約」會議前 5 分鐘，以 Ping 喚醒 vLLM 容器。或者採用 Event-driven，日常完全縮容至 0，而在冷啟動期間前 60 秒的語音，先降級拋給純 CPU 運作的輕量版 CT2 處理 (Warm-up buffer fallback)。

### 盲點四：FMEA 失效分析的「邊緣計算硬著陸」
* **盲點描述**：對於方案 C (Edge WebGPU)，假裝所有使用者的瀏覽器都能正常載入模型是一種倖存者偏差。如果使用者的記憶體不足，前端會直接當機，服務等於完全癱瘓。
* **解決方案 (架構重塑)**：實作 **「漸進式增強 (Progressive Enhancement)」的動態路由**。前端載入時，先默默執行一段環境偵測腳本 (`navigator.gpu` 存在與否、可用 VRAM 探測)。若條件不符，自動降級走 Fallback WebSocket 路線，將端點切換回方案 A 或方案 B 的雲端伺服器，讓終端使用者「無感」地體驗服務。

## 🎯 總結與決策矩陣 (Decision Matrix)

我們將這三者的雷達圖以 第一性原理 的目的（穩定、成本、品質）作為裁決：

| 評量維度 (1-5分) | 方案 A (CT2 堆疊) | 方案 B (MLA+vLLM) | 方案 C (Edge-Cloud) |
|---|:---:|:---:|:---:|
| **開發時程與風險** | **5 (馬上能用)** | 1 (重度開發風險) | 2 (前端生態不成熟) |
| **品質精確度 (WER)** | 5 (滿血模型) | 5 (滿血模型) | 2 (嚴重降維與幻覺) |
| **伺服器擴展成本** | 2 (疊加昂貴) | **5 (極度省錢)** | **5 (不須成本)** |
| **高負載(100場)穩定度** | **4** (多台分散風險) | 2 (OOM 壓力極端) | **5** (去中心化運算) |

### 🏆 架構師最終建議路徑 (Architect's Final Recommendation)

基於 Model-thinking 中的「**階段演化論 (Evolutionary Architecture)**」：

1. **短期護城河 (Phase 1, 0~3 個月)**：
   絕對優先採用 **方案 A (CT2 Faster-Whisper)** 上線。利用成熟的 Cloud Run Scale-out，換取營運初期最珍貴的「穩定度」與「高辨識準確率」。前期日均可能只有不到 10 場，完全負擔得起。
2. **中期利潤極大化 (Phase 2, 當跨越日均 50 場門檻)**：
   背景秘密啟動 **方案 B (Whisper MLA + vLLM)** 的研發，作為壓低維運成本的殺手鐧。當高併發的帳單開始讓利潤流失時，以 Canary Deployment (藍綠佈署) 抽出 10% 流量至 vLLM 測試其 PagedAttention 穩定性。
3. **終極形態 (Phase 3, 長期去中心化戰略)**：
   只為 Premium / 企業級 用戶保留雲端高品質 ASR，同時釋出 **方案 C (Edge-WebGPU)** 版本給免費免登入用戶，利用「用戶設備幫你算出文字再送回平台給 LLM 處理」的商業模式，實現免費流量無限吸納的終極護城河。
