## 會議詳情頁路由統一（方案 2）：消除「列表進入 vs 重新整理」畫面不一致

### 問題（使用者回報）
會議詳情頁「從會議列表進入」與「重新整理（或分享連結直接進入）」呈現的畫面不同。

### 根因（第一性原理盤點，附程式碼證據）
詳情頁其實有**兩條不同的渲染路徑**：

- **列表進入** → `dashboard/page.tsx` 的 `handleViewDetail` 用 `history.pushState`
  改 URL，但**留在 dashboard SPA 內**，以內嵌 `<DetailView>` 顯示，並傳入完整
  callback（regenerate / delete / rename）。
- **重新整理 / 分享連結** → Next 載入 deep-link 路由
  `dashboard/meetings/[meeting_id]/page.tsx`，該頁**刻意只傳** `meeting / onBack /
  onReportThisMeeting`，註解明寫「deep-link 頁暫不暴露 regenerate/delete」。

`DetailView` 的操作全部以「optional callback 是否存在」條件渲染
（`canRegenerate = … && onRegenerateSummary`、`{onDelete && …}`、`{onRename && …}`、
`{onRegenerateTranscript && …}`），故 deep-link 頁缺這些 prop → 重新整理後
**重新生成摘要 / 重新生成逐字稿 / 刪除 / 更名全部消失**，且該頁無 `useMeetingPolling`
→ processing 會議不會自動更新。這就是使用者看到的「畫面不同」。

| 功能 | 列表進入 | 重新整理（舊） |
|---|:--:|:--:|
| 重新生成摘要（換模板） | ✅ | ❌ |
| 重新生成逐字稿 | ✅ | ❌ |
| 刪除 / 更名 | ✅ | ❌ |
| processing 自動輪詢更新 | ✅ | ❌ |

### 解法（方案 2：單一真相來源）
讓 deep-link 路由成為詳情頁的唯一渲染來源，列表點擊改走真路由。

**前端變更**
- `dashboard/page.tsx`
  - `handleViewDetail`：由 `history.pushState` + SPA 內嵌 DetailView 改為
    `router.push('/dashboard/meetings/${id}')`（新增 `useRouter`）。
  - 跨會議引用（cross-meeting ref）與列表點擊皆共用同一入口，行為一致。
- `dashboard/meetings/[meeting_id]/page.tsx`（改為自給自足）
  - 復用既有 hooks：`useMeetings`（deleteMeeting / fetchMeetings）、
    `useSummary`（regenerate summary / transcript）、`useMeetingPolling`
    （processing / pending 自動輪詢，terminal 時以完整版含逐字稿覆蓋）。
  - 補齊完整操作：重新生成摘要 / 重新生成逐字稿 / 更名（`api.renameMeeting`）/
    刪除（`ConfirmDialog` → `deleteMeeting` → `router.push('/dashboard')`）/ 回報。
  - fetch effect 改包進 async function（修 `react-hooks/set-state-in-effect`）。

**權衡**：列表點擊由「即時顯示 list metadata」改為「短暫 loading 後由 detail
endpoint 取完整資料」。以一致性換取單一真相來源，符合方案 2 取捨。

## 專有名詞 LLM 二次修正 + ASR hotword 注入 + 詳情頁入口優化（P0-1/2/3 + UX）

### 背景（源自會議 f71598b8 歷程盤點）
盤點「數位轉型交流會20260703」發現使用者手動輸入 **53 筆會議級 + 4 筆全域**專有名詞
修正、19 個說話者標籤、重生摘要 **7 次**。第一性原理找出三個結構性缺陷：

1. **ASR 端零 biasing**：`get_whisper_prompt()` 有定義但批次上傳流水線（GPU `/asr/refine`）
   從未把 hotword 餵進 Whisper；且 Whisper `initial_prompt` 硬上限 224 token
   （官方：448 context 的一半），CJK 頂多 ~30–50 詞。
2. **無腦 str.replace**：`apply_glossary_correction` 用 Python `.replace()`，context-blind
   會誤傷（如「油桶→郵筒」連正確的油桶也換），且抓不到未登記的長尾錯誤。
3. **摘要不跟隨**：修詞只改 segment、不改 `summary_json` → 使用者被迫整份重生（7 次的主因）。

### P0-1：把專有名詞餵進 ASR（跨 backend + GPU 服務）
- `tasks.py get_whisper_prompt()`：新增 **224-token 字元預算**與優先序（會議級優先 →
  user 級依 usage_count），打包到預算上限；避免超限被 Whisper 靜默截斷。
- `tasks.py`：平行 ASR（`/asr/refine` per-chunk）與單檔路徑的 payload 都帶 `initial_prompt`。
- `gpu_service/main.py`：`ASRRefineRequest` 新增 `initial_prompt`；`_run_asr_processing`
  轉傳給 `transcribe_with_diarization`。
- `offline_asr.py`：`transcribe_with_diarization` / `_transcribe_sync` 穿透 `initial_prompt`
  給 faster-whisper `_model.transcribe(initial_prompt=...)`。

### P0-2：LLM context-aware 修正取代無腦 replace
- `llm_utils.correct_segments_glossary_llm()`：只送「含 wrong 詞」的候選 segment（分批
  GLOSSARY_LLM_BATCH=60），LLM 只在語意確為該詞時替換；回傳僅被改動的段落。
- `tasks.apply_glossary_correction()`：改走 LLM 路徑（env `GLOSSARY_LLM_CORRECTION`
  預設 true），**任何失敗回退 deterministic**（`_apply_glossary_deterministic`）。

### P0-3：詞修後 LLM patch 摘要，免整份重生
- `llm_utils.patch_summary_glossary_llm()`：沿用 relabel 的 JSON 結構驗證（頂層鍵集合
  不變才接受），只改專有名詞字串。
- `tasks.patch_summary_after_correction()`：修完 segment 後就地 patch `summary_json`
  （LLM，fallback：對 JSON 字串做安全 term-swap + 可解析/鍵集合守門）。

### 前端：詳情頁專有名詞入口優化
- **上方入口**（詳情頁置頂 sticky header）：新增「專有名詞」按鈕（BookMarked），
  只在已完成會議顯示。**經確認妥當**——置頂區域是既有操作列所在，可見性最高，
  且浮窗 defaultPosition y=180 已避開 header，不會重演 07-07 遮擋迴歸。
- **下方入口**（逐字稿區）：由舊的灰底小字「顯示/隱藏詞彙」改為與上方**同一共用樣式**
  `glossaryBtnClass`（brand 色、active 實心、更醒目）。
- **首訪主動提示**：首次進入已完成會議跳出「需要修正專有名詞嗎？」callout
  （新增/暫不需要）；以 `localStorage: meetchi:glossary-prompt-seen:{id}` 記錄，
  **第二次進入同一會議不再詢問**。

## 新手引導：讓使用者知道「轉錄完成後還要審稿＋修正專有名詞」

### 問題（第一性原理）
新手的心智模型是「AI 轉錄完 = 結束」，不知道還有：**審稿 → 新增專有名詞 → 按套用修正**。

互相競爭、`套用修正`按鈕新手不知道要按。

### 解法（MECE，按時間軸攔截注意力）
四個攔截點一次到位：

1. **完成通知文案**（`dashboard/page.tsx`）：會議完成 toast 由「點查看即可進入詳情頁」
   改為「建議進入詳情頁**審稿，並修正專有名詞**讓內容更準確」。
2. **合併成「完成後三步」引導卡**（`DetailView`）：移除與 CS-6 競爭的獨立 glossary
   banner；已完成會議首次進入時顯示帶編號的 3 步驟卡（①審稿 ②新增專有名詞 ③按套用

   只在**非完成**狀態顯示，避免雙卡。
3. **審稿空狀態提示**（`DetailView` 逐字稿區）：展開逐字稿時顯示常駐提示列
   「發現錯字 → 按專有名詞新增 → 按套用修正，逐字稿與摘要會一起更新」，內含可直接
   開面板的連結。
4. **套用鈕高亮**（`FloatingGlossaryPanel`）：新增/刪除詞後標記 `needsApply`，讓
   「套用修正」按鈕轉為 brand 實心 + pulse + 輔助字「新增後記得按此套用」；套用成功

## 上傳止血：分塊上傳靜默凍結（孤兒 PENDING）根治

### 事件（管理者帳號實測）
管理者上傳「數位轉型交流會20260703」(id 5776c152)，15 塊分塊上傳僅 chunk 0–4
成功即停滯，狀態凍結在 PENDING、0 task_status、0 段落、~6 分鐘零活動。與
2026-07-06 devlog 記載的根因②完全吻合。

### 根因（第一性原理，附程式碼證據）
`api.ts chunkedUpload.uploadChunkOnce` 的 XHR **完全沒有 `xhr.timeout`/`ontimeout`**。
企業透明 Proxy「hold 住連線但不回應」時，XHR 既不 onload 也不 onerror →
Promise 永久 pending → `uploadChunkWithRetry` 的重試迴圈 `await` 卡死、永不觸發 →
整個上傳靜默凍結。因 compose 只在最後一塊執行 → 音檔永不完成 → 孤兒 PENDING。

### 修復
**前端（主修復，根治靜默凍結）** `src/lib/api.ts`：
- `uploadChunkOnce` 加**兩道停滯防線**：
  1. `xhr.timeout = 120s` 硬上限 + `ontimeout` → reject 觸發重試。
  2. **進度閒置看門狗**：追蹤最後一次 `upload.onprogress` 時間，30s 無任何 bytes
     前進即 `xhr.abort()` + reject（比硬上限更快抓到「連線 hold 住但零傳輸」）。
- 重試改良：`MAX_RETRIES` 3→5；退避改指數 + 抖動（1/2/4/8s +0–500ms jitter），
  避免多塊同時重試打爆 proxy；AbortError 不觸發重試。

**後端（安全網，孤兒清理）** `app/routes/ops_admin.py`：
- ops 會議清單的 stuck 偵測納入**孤兒 PENDING**（PENDING 超過 30 分且 0 段落）。
- 新增 `POST /api/v1/ops/cleanup-orphan-uploads`（admin，`dry_run` 預設 true）：
  掃描逾時孤兒 PENDING → 標記 FAILED 附可讀原因「上傳未完成…請重新上傳」，
  讓使用者不再看到永久「排隊中」。可由 Cloud Scheduler 定期呼叫（dry_run=false）
  自動化 TTL 清理。**非刪除**，可還原/重試。

- 後端：`py_compile` + import OK。

## UAT 安全加固：關閉「未認證即回 mock admin」+ callback 驗證（P0–P2）

### 🔴 調查結論（附實證）
後端 API 對整個網際網路**公開且形同無認證**：真正無憑證的請求（`env -i curl`）
`GET /api/v1/ops/overview` 回 **200** 並洩漏生產資料（8 用戶/29 會議/成本 $172.8）。
根因：Cloud Run 未設 `AUTH_REQUIRED` → 程式碼預設 false → `get_current_user` 對所有
請求回 mock `dev@example.com` → `require_admin` 解析成真 admin。認證被完全繞過。
`ingress=all` 放大暴露面。

### 修復（P0–P2，UAT）
**P0 `AUTH_REQUIRED=true`**（增量 gcloud，非 --set-*）。
- 前置修復（否則會打斷 UAT）：`api` 是 module singleton，先前只有 dashboard/page.tsx
  setToken → 直接進入 deep-link 詳情頁（方案2 後的主入口）token 為 null。新增
  `ApiTokenSync` 掛在 dashboard layout，全域佈線 idToken 到 api client。
**P1a `AUTH_ALLOWED_DOMAIN=mail.chimei.com.tw`**（正式 OAuth 限網域；UAT token 仍 bypass）。
**P1b callback 驗證**：`/callbacks/asr-done` 先前無認證（任意人可偽造 ASR 結果）。
新增 `verify_asr_callback` 驗證 GPU 已附的 Google OIDC token（驗簽章 + aud 比對
BACKEND_PUBLIC_URL + SA email fallback），以 `CALLBACK_AUTH_REQUIRED` 閘控。
**P2 ingress**：**無法**收成 internal——前端 `NEXT_PUBLIC_API_URL` 是瀏覽器直連後端，
收窄會打斷所有瀏覽器呼叫。UAT 維持 `ingress=all`，以 app-layer auth（P0）為控制點；
PROD 才以 VPC + LB + IAP + internal ingress 補網路層。

  亂 token→401；health 仍 200。

後端同步收結果），callback 僅影響 <20min 短音檔。

且 cloudrun.tf 內 URL 仍是舊 `705495828555`、缺 AUTH_SECRET/ADMIN/SMTP/GPU 等線上 env。
**直接 `terraform apply` 會以不完整 env 覆蓋線上、造成中斷**。故本次 env 變更以增量 gcloud
套用（agents.md 允許的 --update-env-vars，非 --set-*），並回填 cloudrun.tf 作 SoT 文件

## 迴歸修復：AUTH_REQUIRED 後模板頁「載入失敗」（token 時序競態）

### 問題（使用者回報）

## 緊急修復：非管理者（UAT）帳號登入後全部 401「無法連線」

## Whisper 幻覺抑制（穩健型）+ 溫度 fallback 修正 — chunk_000 基準 A/B

### 問題
uat03 會議前 ~4 分鐘（會前環境雜訊/閒聊/重疊語音）被 Whisper 強制解碼成連鎖幻覺

### 根因（第一性原理）
1. `condition_on_previous_text` 預設 True → 幻覺跨段自我延續（「雙子座的元素」長迴圈）。
2. 未設 compression_ratio / log_prob / no_speech 門檻 → 低品質輸出不被濾除。
3. VAD 寬鬆（min_silence 500ms）→ 短暫低訊號被當語音。

### 修復（兩階段，均以 chunk_000 為基準比對）
**階段 1**（robust 參數）：condition_on_previous_text=False、compression_ratio=2.4、
log_prob=-1.0、no_speech=0.6、VAD min_silence 500→1000ms + threshold=0.5。
→ 斷開跨段延續，但殘留 `現在現在現在...`×39 的段內重複。
**根因追加發現**：誤設 `temperature=0.0` **關閉了 faster-whisper 的溫度 fallback**——
compression_ratio / log_prob 門檻只在 fallback 溫度用盡時才真正丟棄壞段落，
單一溫度使門檻失效。
**階段 2**（修正）：改用溫度 fallback ladder `(0.0..1.0)`，門檻恢復作用。

### A/B 結果（前 10 分鐘，chunk_000 基準）
- v1（temp=0.0）：04:16 出現 `現在 現在 現在...`（39 次連續重複）。
- **v2（temp fallback）：同位置縮為 `現在先在這一塊`——災難級重複迴圈消除。**
- 跨段幻覺延續已斷（無 v0 那種長迴圈）。

### 誠實揭露（殘留與界線）
- 前 ~4 分鐘音檔本身為**會前低品質語音**（重疊/雜訊/麥克風差），仍有零星短重複
