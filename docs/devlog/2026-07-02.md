# GPU Cloud Run 費用洩漏修復（meetchi-gpu-asr scale-to-zero）

## 背景 / 問題
帳單分析 6/25–6/28 Cloud Run 成本時，發現 `meetchi-gpu-asr`（NVIDIA L4 + 8 vCPU + 32Gi）
連續 4 天 ~24h/day 保持 **1 個暖實例**，其中 6/26–6/27 **request_count = 0** 卻仍
`instance_count = 1.00`（flat 48h）→ 純燒錢（L4 bundle ~$0.86/hr，4 天約 $85，其中 6/26–6/27
約 $42 為零流量純浪費）。

## 真正根因（非表面 min=1）
- 100% 流量**鎖定**在舊 revision `meetchi-gpu-asr-phaseb-emb`（minScale=**1**，
  建於 2026-06-22T02:54:11Z），該 min=1 讓 1 個 L4 實例 24/7 常駐。
- 較新的 `00073-p6m`（min=0，同日 08:50 建立）雖已部署，但**流量從未切過去**
  → 又是 MeetChi 反覆出現的「流量鎖定」模式（新 revision 部署但流量留在舊 revision）。
- 意圖狀態：Terraform `min_instance_count = var.min_instances`、docs `min_instances = 0`
  → 設計本就是 scale-to-zero。live 的 min=1 為組態漂移。
  （`terraform/cloudrun.tf` 的 `gpu_asr` 區塊為註解狀態，非 TF 管理，正規管理路徑為 gcloud，
   與 DEPLOYMENT.md 一致。）

## 修復（未用任何 --set-* replace-all flag）
1. 以現有 image 重部署，明確 `--min-instances=0`，保留 GPU 硬體：
   `gcloud run deploy meetchi-gpu-asr --image ...@sha256:716a5dec... --min-instances=0
    --gpu=1 --gpu-type=nvidia-l4 --no-cpu-throttling`
   → 產生新 revision `meetchi-gpu-asr-00074-85q`（無 minScale 註解＝min=0）。
2. **再次命中流量鎖定**：deploy 後流量仍 100% 在 `phaseb-emb`。手動切流量：
   `gcloud run services update-traffic meetchi-gpu-asr --to-revisions meetchi-gpu-asr-00074-85q=100`
   → 流量 100% → `00074-85q`，舊 min=1 暖實例開始排空。

## 驗證狀態（物理證據，時間 UTC+8）
- ✅ 冷啟健康檢查：切流量後 `GET /health` → **200**（/healthz、/ 為 404，屬正常路由）
  → 證明 min=0 冷啟後 ASR 服務仍可用。
- ✅ scale-to-zero 生效（Cloud Monitoring `container/instance_count`，service=meetchi-gpu-asr）：
  - 11:20–11:25 active=1（切換前暖實例＋健康檢查冷啟）
  - **11:26 起 active=0 連續 16+ 分鐘（至 11:42 仍為 0）** → 閒置後成功降到零。
- 效果：閒置時段不再燒 L4，預期消除 6/26–6/27 型態的零流量浪費（~$42/相同天數區間）。

## 後續注意
- min=0 常駐 → 冷啟延遲回歸（L4 冷啟較久）。6 月曾有「營業時段暖機」自動化
  （00:00 UTC min=1 / 09:00 UTC min=0，約 6/22 後停止；目前無 Cloud Scheduler job）。
  若需營業時段暖機以降低冷啟，建議以 Cloud Scheduler 排程切 min，而非常駐 min=1。**待與用戶確認再實作**。

# 設計系統修復（a+b+c）：brand-accent 死 token + tailwind.config.ts 未載入 + 稽核文件回註

## 起因
盤點 `docs/design/` 四份設計文件並比對實作，發現三個實質問題（a/b/c）。

## (a) brand-accent 死 token（實際 UI bug）
- **問題**：DDG-CHIMEI.md 宣稱 `brand-accent` 已全 codebase 移除、再用會 build error——兩點都錯。
  `TemplateGallery.tsx` `COLOR_MAP` 仍用 `bg-brand-accent`，後端 `template_engine.py` 的
  `tpl-rd` / `tpl-standup` 系統模板 `color` 仍回傳 `brand-accent`。而 `brand-accent` 在 v4 `@theme`
  **未定義** → 不是 build error，而是**靜默渲染成無色**。
- **修法**：
  - 前端 `TemplateGallery.tsx`：`COLOR_MAP` `brand-accent` → `brand-violet`；並加 legacy normalization
    `color === 'brand-accent' ? 'brand-violet' : color` 相容既有資料。
  - 後端 `template_engine.py`：`tpl-rd` / `tpl-standup` 的 `color` → `brand-violet`（此為 API 即時回傳的
    `SYSTEM_TEMPLATES`，非 DB 種子，故需重部署後端才會生效）。

## (b) tailwind.config.ts 在 v4 下未被載入（多項功能靜默壞掉）
- **根因**：本專案為 Tailwind v4（`@tailwindcss/postcss` + `@import "tailwindcss"` + `@theme`），
  globals.css **無 `@config`**，故 `tailwind.config.ts` **完全未被載入**。物理證據：base build 編譯後 CSS
  `grep` 不到 `ease-brand` / `animate-indeterminate` / `prose` / `animate-in`（count=0）。
- **連帶靜默壞掉**：
  - `prose`（`@tailwindcss/typography`）→ RAG 對話 markdown 無樣式
  - `animate-in`/`fade-in`/`slide-in-from-*`/`zoom-in-95`（`tailwindcss-animate`）→ 多處進場動畫不播；
    globals.css 手工補的相容 utility 只覆蓋極少數變體（連 `slide-in-from-top-2`/`zoom-in-95` 都沒有）
  - `ease-brand`（MeetingCard hover）、`animate-indeterminate`（MeetingCard 進度條）→ 失效
  - `dark:` 變體（config `darkMode:"class"` 未載入）→ v4 fallback 成 `prefers-color-scheme` media，
    **無視 .dark toggle**（`dark:text-white` / `dark:prose-invert` 實際不跟切換）
- **修法（v4 原生，拒絕重造輪子）**：於 `globals.css`
  - `@plugin "@tailwindcss/typography";` + `@plugin "tailwindcss-animate";`（正確載入既有依賴）
  - `@custom-variant dark (&:where(.dark, .dark *, [data-theme="dark"], [data-theme="dark"] *));`（class-based dark）
  - `@theme` 補 `--ease-brand` / `--ease-brand-out` / `--animate-indeterminate` + `@keyframes indeterminate`
  - 移除不完整的手工 animate 相容層（`.animate-in`/`.fade-in`/`.slide-in-*` + fadeIn/slideIn keyframes），
    避免與 plugin 同名衝突
  - **刪除死的 `tailwind.config.ts`**
