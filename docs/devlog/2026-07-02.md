## 環境備忘：gcloud「Regional Access Boundary」雜訊

- 現象：Cloud Shell（devshell）環境下，每次 gcloud 指令都先噴
  `Regional Access Boundary HTTP request failed ... Account not found for email: 6f70026545|jerry_tai@mail.chimei.com.tw (404)`，
  之後仍 fallback 成功。
- 根因：組織層級 Regional Access Boundary（資料落地）強制 + devshell 代理憑證把帳號格式化為
  `<clientId>|<email>`，RAB 解析端點不認得而回 404。**非致命**——list / describe / deploy / update-traffic 皆正常。
- `regional/endpoint_mode` 未設、無 env override，故非 endpoint mode 造成。
- 真正消除：需互動式 `gcloud auth login`（瀏覽器）換掉 devshell 代理憑證為第一方使用者憑證；
  無頭工具無法觸發，須在 Cloud Shell 終端手動執行。目前不影響部署，暫維持現狀。

# GPU Cloud Run 費用洩漏修復（meetchi-gpu-asr scale-to-zero）

## 背景 / 問題
帳單分析 6/25–6/28 Cloud Run 成本時，發現 `meetchi-gpu-asr`（NVIDIA L4 + 8 vCPU + 32Gi）
連續 4 天 ~24h/day 保持 **1 個暖實例**，其中 6/26–6/27 **request_count = 0** 卻仍
`instance_count = 1.00`（flat 48h）→ 純燒錢（L4 bundle ~$0.86/hr，4 天約 $85，其中 6/26–6/27
約 $42 為零流量純浪費）。

## 真正根因（非表面 min=1）
- 100% 流量**鎖定**在舊 revision `meetchi-gpu-asr-phaseb-emb`（minScale=**1**，
  建於 2026-06-22T02:54:11Z），該 min=1 讓 1 個 L4 實例 24/7 常駐。
- 較新的 `00073-p6m`（min=0，同日 08:50 建立）雖已部署，但**流量從未切過去**
  → 又是 MeetChi 反覆出現的「流量鎖定」模式（新 revision 部署但流量留在舊 revision）。
- 意圖狀態：Terraform `min_instance_count = var.min_instances`、docs `min_instances = 0`
  → 設計本就是 scale-to-zero。live 的 min=1 為組態漂移。
  （`terraform/cloudrun.tf` 的 `gpu_asr` 區塊為註解狀態，非 TF 管理，正規管理路徑為 gcloud，
   與 DEPLOYMENT.md 一致。）

## 修復（未用任何 --set-* replace-all flag）
1. 以現有 image 重部署，明確 `--min-instances=0`，保留 GPU 硬體：
   `gcloud run deploy meetchi-gpu-asr --image ...@sha256:716a5dec... --min-instances=0
    --gpu=1 --gpu-type=nvidia-l4 --no-cpu-throttling`
   → 產生新 revision `meetchi-gpu-asr-00074-85q`（無 minScale 註解＝min=0）。
2. **再次命中流量鎖定**：deploy 後流量仍 100% 在 `phaseb-emb`。手動切流量：
   `gcloud run services update-traffic meetchi-gpu-asr --to-revisions meetchi-gpu-asr-00074-85q=100`
   → 流量 100% → `00074-85q`，舊 min=1 暖實例開始排空。

## 驗證狀態（物理證據，時間 UTC+8）
- ✅ 冷啟健康檢查：切流量後 `GET /health` → **200**（/healthz、/ 為 404，屬正常路由）
  → 證明 min=0 冷啟後 ASR 服務仍可用。
- ✅ scale-to-zero 生效（Cloud Monitoring `container/instance_count`，service=meetchi-gpu-asr）：
  - 11:20–11:25 active=1（切換前暖實例＋健康檢查冷啟）
  - **11:26 起 active=0 連續 16+ 分鐘（至 11:42 仍為 0）** → 閒置後成功降到零。
- 效果：閒置時段不再燒 L4，預期消除 6/26–6/27 型態的零流量浪費（~$42/相同天數區間）。

## 後續注意
- min=0 常駐 → 冷啟延遲回歸（L4 冷啟較久）。6 月曾有「營業時段暖機」自動化
  （00:00 UTC min=1 / 09:00 UTC min=0，約 6/22 後停止；目前無 Cloud Scheduler job）。
  若需營業時段暖機以降低冷啟，建議以 Cloud Scheduler 排程切 min，而非常駐 min=1。**待與用戶確認再實作**。
- 流量鎖定為反覆模式（frontend + gpu-asr 皆中招）：每次 deploy 後務必手動確認/切 LATEST 流量。
