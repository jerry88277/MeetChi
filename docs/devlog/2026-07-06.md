## 上傳失敗根因分析（第一性原理 + MECE）

使用者上傳 ~120MB「數位轉型交流會」音檔，連兩次瀏覽器分段上傳都在 40–60%（chunk 6–9）附近中斷。

### 根因（兩層疊加）
- **根因①（觸發）**：企業網路封鎖直連 `storage.googleapis.com` → 前端三層 fallback（resumable 直連 → signed PUT 直連 → backend proxy chunk）被迫走最爛的第三層 backend-proxy，位元組繞經 Cloud Run，曝險加倍。
- **根因②（致命放大）**：`api.ts` 的 `uploadChunkOnce` 的 XHR **沒有設 `xhr.timeout`**。企業透明 Proxy 常「hold 住連線但不回應」→ XHR 既不 onload 也不 onerror → Promise 永久 pending → 重試邏輯不會觸發 → 整個上傳靜默凍結在 ~47%，伺服器端零錯誤、會議留下孤兒 PENDING。

### MECE 優化清單（依位元組路徑分層）
A. 客戶端邏輯：XHR timeout+ontimeout、加長退避+抖動、失敗 chunk 不牽連整體、stalled 提示。
B. 網路/Proxy：自訂網域反代 GCS 讓 resumable 直連成為主路徑；縮小 chunk（8→2–4MB）。
C. 協定/續傳：利用已存在的 `parts/NNNNNN` 做「只補缺塊」續傳；跨策略共用進度。
D. 後端合成：streaming 寫入；compose 前先驗證 parts 齊全（缺塊回明確清單而非 500）。
E. GCS：parts lifecycle 自動清理；chunk CRC32C 校驗。
F. 控制面：孤兒 PENDING TTL 掃描清理；enqueue 前再驗 GCS 物件存在。
G. 觀測/UX：upload_id 貫穿前後端；`stream_audio` 對 404 回乾淨 409/425 而非拋例外汙染 log。

（P0 止血：根因②的 XHR timeout；後端 compose 前驗證 + 前端只補缺塊續傳。待實作。）

### #1 等待焦慮 — 轉錄卡住 watchdog（DetailView）
- 問題：使用者被告知 ETA，但超時後畫面仍停在「處理中/排隊中」轉圈，無任何下一步指引（非失敗畫面）。根因是 `useMeetingPolling` 與 processing/pending UI **完全沒有逾時偵測**，會永遠輪詢+轉圈。
- 修法：加只在 pending/processing 運作的計時器，依音檔長度估算預期時間：
  - `pending` 逾 5 分 → 「似乎卡住了，可能上傳未完成」+ 重新整理／立即回報 + 提示重新上傳。
  - `processing` 逾預期 → 軟提示「比預期久，仍在處理中，**不需重新上傳**」。
  - `processing` 逾預期 ×2 → 「可能卡住了，可能網路不穩」+ 重新整理／重新從頭轉錄／立即回報。

### #2 必填欄位醒目提示（FeedbackModal）

## 三項新功能：降噪 / 逐段改說話者 / 摘要同步（本日實作）

依第一性原理與 MECE 拆解使用者情境（VDI 會議室、喇叭播放→筆電內建麥克風錄音，機型 ASUS ExpertBook B1402CVA）。

### #1 後端降噪（可行性結論 + 保守實作，flag-gated，**暫不部署 GPU**）
- 可行性：✅ 可行但須**保守**。研究結論：喇叭→麥克風情境的殘響/回聲比雜訊更傷 ASR；**輕度**降噪（highpass 去低頻嗡聲 + 溫和 `afftdn` + `loudnorm` 響度正規化）在低訊噪比時有助 ASR 且保留語音資訊；**重度**降噪反而吃掉子音、產生偽影，降低 Whisper 準確率。
- 實作：`apps/backend/gpu_service/main.py` 新增 `_denoise_audio()`，於音檔下載後、轉錄前插入（`_run_asr_processing`）。以 `ENABLE_DENOISE`（預設 false）開關，濾鏈可由 `DENOISE_AF` 覆寫以利 A/B。ffmpeg 失敗自動回退原檔，確保不劣化。
- **決策**：依使用者指示先實作、flag 關閉、**先不重新部署 GPU**（GPU 成本敏感，見 checkpoint 001）。底部播放器改放降噪後音檔一併等 GPU 部署時處理。

### #2 逐段重指派說話者（per-segment speaker reassignment）
- 問題：ASR 把不同說話者的話錯歸為同一人；聚合視圖無法單獨修正。
- 後端：`PATCH /api/v1/meetings/{id}/segments/speakers`，body `{ updates: { segment_id: 新原始標籤 } }`，更新 `TranscriptSegment.speaker`。
- 前端：`transform.ts` 新增未聚合 `rawSegments`（帶 id）；DetailView 逐字稿區新增「逐段編輯說話者」模式，每段下拉選擇既有說話者，追蹤待存變更，儲存後重新載入取得最新聚合。

### #3 摘要同步說話者（hybrid：LLM 快掃重貼 + 建議重生）
- 問題：更新說話者後摘要未同步。
- 後端：`POST /api/v1/meetings/{id}/resync-summary-speakers`。`llm_utils.relabel_summary_speakers()` 用 Gemini **僅修正摘要中的說話者名稱引用**（保留其餘內容與 JSON 結構，頂層鍵不一致則拒絕採用）；並依啟發式（逐字稿實際說話者集合 vs 摘要 `speaker_contributions` 引用集合）判斷是否**建議整份重生**。變更前備份 `SummaryVersion`。
- 前端：DetailView 新增「更新摘要（同步說話者）」鈕；結果面板顯示已同步/無需修改/建議重生，並提供「重新整理查看」與「整份重生摘要」動作。
