## 上傳失敗根因分析（第一性原理 + MECE）

使用者上傳 ~120MB「數位轉型交流會」音檔，連兩次瀏覽器分段上傳都在 40–60%（chunk 6–9）附近中斷。

### 根因（兩層疊加）
- **根因①（觸發）**：企業網路封鎖直連 `storage.googleapis.com` → 前端三層 fallback（resumable 直連 → signed PUT 直連 → backend proxy chunk）被迫走最爛的第三層 backend-proxy，位元組繞經 Cloud Run，曝險加倍。
- **根因②（致命放大）**：`api.ts` 的 `uploadChunkOnce` 的 XHR **沒有設 `xhr.timeout`**。企業透明 Proxy 常「hold 住連線但不回應」→ XHR 既不 onload 也不 onerror → Promise 永久 pending → 重試邏輯不會觸發 → 整個上傳靜默凍結在 ~47%，伺服器端零錯誤、會議留下孤兒 PENDING。

### MECE 優化清單（依位元組路徑分層）
A. 客戶端邏輯：XHR timeout+ontimeout、加長退避+抖動、失敗 chunk 不牽連整體、stalled 提示。
B. 網路/Proxy：自訂網域反代 GCS 讓 resumable 直連成為主路徑；縮小 chunk（8→2–4MB）。
C. 協定/續傳：利用已存在的 `parts/NNNNNN` 做「只補缺塊」續傳；跨策略共用進度。
D. 後端合成：streaming 寫入；compose 前先驗證 parts 齊全（缺塊回明確清單而非 500）。
E. GCS：parts lifecycle 自動清理；chunk CRC32C 校驗。
F. 控制面：孤兒 PENDING TTL 掃描清理；enqueue 前再驗 GCS 物件存在。
G. 觀測/UX：upload_id 貫穿前後端；`stream_audio` 對 404 回乾淨 409/425 而非拋例外汙染 log。

（P0 止血：根因②的 XHR timeout；後端 compose 前驗證 + 前端只補缺塊續傳。待實作。）

### #1 等待焦慮 — 轉錄卡住 watchdog（DetailView）
- 問題：使用者被告知 ETA，但超時後畫面仍停在「處理中/排隊中」轉圈，無任何下一步指引（非失敗畫面）。根因是 `useMeetingPolling` 與 processing/pending UI **完全沒有逾時偵測**，會永遠輪詢+轉圈。
- 修法：加只在 pending/processing 運作的計時器，依音檔長度估算預期時間：
  - `pending` 逾 5 分 → 「似乎卡住了，可能上傳未完成」+ 重新整理／立即回報 + 提示重新上傳。
  - `processing` 逾預期 → 軟提示「比預期久，仍在處理中，**不需重新上傳**」。
  - `processing` 逾預期 ×2 → 「可能卡住了，可能網路不穩」+ 重新整理／重新從頭轉錄／立即回報。

### #2 必填欄位醒目提示（FeedbackModal）
- 問題：送出鈕在未填時 disabled，使用者不知道**缺哪一項**。
- 修法：啟用按鈕；點擊若無效 → 缺漏必填欄位（問題類型／描述／嚴重程度）轉紅框+提示，並 toast 列出缺項。欄位填妥後紅框自動消失。

### #3 前端輸入延遲（每字慢 0.5–1s）
- 調查（explore agent）：FeedbackModal 是**未 memo 的內聯子元件**，位於龐大的 dashboard page 樹中，隨 dashboard 每次重繪（10s 輪詢等）連帶重繪；且全域 console hook 為主執行緒稅。
- 修法：`React.memo` + `createPortal` 到 `document.body` 隔離 modal；`page.tsx` 以 `useCallback` 穩定 `onClose`；`DashboardView` 搜尋過濾改 `useMemo`。
- 後續建議：console hook 改為僅在錯誤路徑收集（降低全域 main-thread 稅）；MeetingCard `React.memo`；搜尋 debounce。

### #4 說話者標籤 Enter 存檔（DetailView）
- 問題：名稱／角色輸入框無鍵盤處理，只能點勾勾存檔。
- 修法：兩個輸入框加 `onKeyDown`，Enter → `handleSaveSpeaker`、Escape → 取消。
