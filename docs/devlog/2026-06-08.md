## 一、已解決問題

### 1-A. RAG 跨會議知識庫無法取得使用者會議記錄（根本原因）

**現象**：任何查詢都無結果，RAG 看起來像是回傳寫死的空回答。  
**根本原因**：`main.py` 啟動時補齊孤兒會議（`meeting_participants` 缺席記錄）的 migration，原本硬寫 `test@company.com` 作為 participant。所有真實使用者（含 `jerry_tai@mail.chimei.com.tw`）因此在 `JOIN meeting_participants ON user_upn` 時永遠匹配不到任何資料。  
**修復**：改為讀取 `meetings.owner_upn` 欄位，以實際上傳者作為 participant 插入。  
**手動補救**：對既有資料執行一次性 backfill API：
```bash
POST /api/v1/admin/backfill-participants?user_upn=jerry_tai@mail.chimei.com.tw
# → 回傳：3 meetings linked
```
此 API 為冪等操作，可重複執行。  
**部署**：Backend rev 00014，tag `20260608-fixes`。

---

### 1-B. 歷史對話紀錄在重新進入頁面後消失（RAG Chat History）

**現象**：從側邊欄重新進入跨會議知識庫，對話狀態完全重置。  
**根本原因**：`ChatPanel.tsx` 的 `messages` state 純粹存在記憶體，路由切換即銷毀。  
**修復**：使用 `sessionStorage`，key = `rag_messages_${userUpn}`，確保：
- 帳號 A / 帳號 B 各自隔離
- 頁面重新整理後仍可恢復
- 切換帳號自動清除（key 不同）  
**已知限制**：`sessionStorage` 以瀏覽器標籤為單位，關閉標籤後清除。正式持久化需後端 DB（列為後續 backlog）。

## 二、新功能

### 2-A. Greeting Feature（五星飯店迎賓體驗）Backend

**目標**：使用者進入跨會議知識庫時，AI 以個人化迎賓語問候，展示：
- 使用者顯示名稱（upn 前缀 or 姓名）
- 可存取的會議數量 / 總時長
- 最常出現的討論主題 pills
- 待辦行動項目數量
- 建議提問 chips

**架構決策**：採 Approach A（DB + template）：
- **不呼叫 LLM** → 零延遲、零費用、100% 可預期輸出
- 資料來源：`meetings`、`meeting_participants`、`meeting_action_items` 三張表
- 主題萃取邏輯：章節標題 × 2 權重 + 說話人貢獻主題 × 1 + 會議標題 × 1，取 top-5
- 全程 try/except 包覆，永遠回傳 fallback payload，不拋例外

**新增檔案**：
- `apps/backend/app/services/rag_greeting.py`（278 行，9 個函式）
- `apps/backend/app/services/__init__.py`

**新增 API**：`GET /api/v1/rag/greeting?user_upn={upn}`  
- 驗證 `@` 符號，normalise lowercase
- 回傳 `RagGreetingResponse` Pydantic model

**依賴新 Model**：`MeetingActionItem`（`models.py`）+ `_sync_action_items()`（`tasks.py`）
- 在 `generate_summary_core()` 完成摘要 DB commit 之後，自動同步 action items 至 `meeting_action_items` 表
- Delete-then-reinsert（冪等），wrapped in try/except（非致命）

**前端基礎**：`lib/api.ts` 已新增 `getRagGreeting()` 方法及 TS types。  
**⚠️ 待完成**：`ChatPanel.tsx` Greeting Card UI（g6-frontend-ui，尚未實作）
