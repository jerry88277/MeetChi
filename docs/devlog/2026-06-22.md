# DevLog 2026-06-22: Cloud Run 成本調查與降本措施

## 背景

帳單顯示 Cloud Run 每日費用穩定在 ~$80 USD，遠超預期。本次針對實際 live 配置進行深度調查，找出成本根因並執行第一輪降本措施。

## 調查發現

### 成本結構拆解（每日）

| 服務 | 配置 | 日成本 | 佔比 |
|------|------|--------|------|
| **meetchi-gpu-asr** | 8 vCPU / 32 GiB / 1× L4 GPU, minScale=1 | $69.81 | 87% |
| meetchi-backend | 2 vCPU / 4 GiB, cpu-throttling=false | ~$5.01 | 6% |
| meetchi-frontend | 1 vCPU / 512 Mi, cpu_idle=true | ~$0.50 | <1% |
| 其他（LB, SQL, Storage） | — | ~$2-5 | ~5% |
| **合計** | | **~$80/天** | |

### GPU ASR 成本細項

| 計費項目 | 單價 | 日成本 |
|---------|------|--------|
| GPU (L4) | $0.000536/GPU-sec | $46.31 |

### 發現的問題

#### 🔴 問題 1: GPU minScale=1 導致 24/7 計費
- `minScale=1` 使 L4 GPU instance 永不關閉
- 即使凌晨零流量也持續燒 $69.81/天
- GPU 服務強制 `cpu-throttling=false`（always-allocated），無法降為 request-based

#### 🔴 問題 2: 殭屍 Tagged Revisions
- `v12` (meetchi-gpu-asr-00016-xen) 和 `v13` (meetchi-gpu-asr-00017-woq) 兩個舊版 revision 仍有 tag
- 每個 tagged revision 帶有 `minScale=1`
- Tagged revision 有獨立 URL，GCP 可能為其保持 instance 活躍
- 潛在額外成本：每個 +$69.81/天

#### 🟠 問題 3: Terraform vs Live 配置偏移
| 參數 | Terraform 宣告 | 實際 Live |
|------|---------------|-----------|
| maxScale | 1 | **15** |
| concurrency | 1 | **2** |
| Tagged revisions | 無 | **v12, v13** |

#### 🟡 問題 4: Backend always-allocated CPU
- `cpu-throttling=false` = instance 閒置時仍按全 CPU 計費
- Backend 每 ~25 分鐘 autoscale 新 instance → 幾乎持續活躍
- 日成本 ~$5（若改 request-based 可降至 ~$2）

## 執行的變更

### 變更 1: 移除殭屍 Tagged Revisions ✅
```bash
gcloud run services update-traffic meetchi-gpu-asr \
  --remove-tags=v12,v13 --region=asia-southeast1
```
- 時間：2026-06-22 16:52 (UTC+8)
- 結果：僅剩 `phaseb-emb` 服務 100% 流量

### 變更 2: GPU ASR minScale=1 → 0 ✅
```bash
gcloud run services update meetchi-gpu-asr \
  --min-instances=0 --region=asia-southeast1
```
- 時間：2026-06-22 16:53 (UTC+8)
- 影響：無流量時 instance 完全關閉，首次請求 cold start 約 45-60 秒
- 目的：純按使用量計費，避免 24/7 空燒

### 變更 3: 刪除 Cloud Scheduler 排程 ✅
```bash
gcloud scheduler jobs delete gpu-asr-schedule-up --location=asia-southeast1
gcloud scheduler jobs delete gpu-asr-schedule-down --location=asia-southeast1
```
- 時間：2026-06-22 16:54 (UTC+8)
- 原排程：週一～五 08:00 scale-up、17:00 scale-down
- 理由：改 minScale=0 後排程已無意義（不再需要定時開關機）

### 未變更: maxScale 維持 15
- 原因：確保多人同時上傳時能 autoscale 分流，避免排隊等候過久
- 目前另一 session 正在進行壓力測試驗證

### 未變更: Backend cpu-throttling
- 保持 `cpu-throttling=false`
- 需進一步確認 backend 是否有 background task / WebSocket 依賴 always-allocated CPU
- 省下的 ~$3/天 vs 潛在穩定性風險，暫不冒險

## 風險與回退方案

若 cold start 嚴重影響使用體驗：
```bash
# 回退：恢復 minScale=1
gcloud run services update meetchi-gpu-asr \
  --min-instances=1 --region=asia-southeast1
```

折衷方案：搭配 Cloud Scheduler 在工作日 08:00 暖機：
```bash
gcloud scheduler jobs create http gpu-asr-warmup \
  --schedule="0 8 * * 1-5" \
  --uri="https://meetchi-gpu-asr-atro34poxq-as.a.run.app/health" \
  --location=asia-southeast1
```
