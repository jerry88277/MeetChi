# Devlog 2026-07-03（UTC+8 台北時間）

## ChiMemo RAG 對抗式稽核 + 修正優化

### 背景
使用者提供一則 ChiMemo 跨會議問答輸出（實際查詢 meeting
`b4d5503e-0a24-4984-a94e-4c1d87de6c08`），要求以第一性原理 + MECE + 多角色對抗式
稽核找出問題，並實作同意的修正。完整稽核報告：`docs/audits/chimemo-rag-audit-2026-07-03.md`。

### 稽核主要發現（皆附程式碼實證）
1. **BUG#1（HIGH）引用索引錯位**：`routes/rag.py` 送給 LLM 的 `prompt_citations`
   （未合併的 expanded_rows）與回傳前端的 `citations`（合併重排後）順序/數量不同，
   導致答案中的 `[來源N]` 指到錯誤的 citation 卡片 → 分數對不上、點擊跳錯。
2. 三通道（segment / summary / title-match）無會議級去重 → 同一場重複出現。
3. segment 通道無相似度門檻（summary 有 <0.3 skip） → 57% 雜訊也顯示。
4. `used_citation_indices` 後端有算但前端渲染全部 citations → 顯示未引用雜訊。
5. `no_answer` fallback 反而用標題匹配摘要「編答案」，違反 grounding 誠實原則，且含 markdown。
6. 講者名未串接：RAG 取原始 `ts.speaker`（SPEAKER_A），從不 JOIN `meetings.speaker_mappings`。
7. 相似度百分比對使用者無意義（且因 BUG#1 常是錯來源分數）。
8. 待辦「數量」顯示但無勾選/消除機制。

### 已實作修正
**後端 `apps/backend/app/routes/rag.py`**
- **BUG#1 對齊**：移除獨立的 `prompt_citations`，改用與前端「同一份、同順序」的
  `citations` 組 grounded prompt，`[來源N]` 一一對應。
- **Q3 講者真名串接**：新增 `_fetch_speaker_mappings`（批次抓 `meetings.speaker_mappings`）
  與 `_resolve_speaker`（label → display_name（role）；無對應 fallback「講者 N」，
  永不外露 SPEAKER_xx，對齊前端 `SpeakerName.tsx`），於 citation 分群時套用。
- **Fix E 相似度門檻**：`SEGMENT_SIM_FLOOR = 0.35`，剔除低相關段落；標題匹配會議豁免；
  全數被剔則保留最強 1 筆避免空手。
- **Fix F no_answer 誠實化**：title-match fallback 改為明確標示「逐字稿無法直接回答，
