### 問題描述
會議詳細頁面底部的音訊播放器始終顯示 0:00/0:00，無法播放。

### 已嘗試方法（按時間序）

| # | 日期 | 方法 | 解決了 | 未解決 |
|---|------|------|--------|--------|
| 1 | 06-15 | GCS compose 後設定 `Content-Type: audio/mp4` | metadata 為 None 的問題 | 瀏覽器仍無法 parse duration |
| 2 | 06-15 | `<audio preload="metadata" crossOrigin="anonymous">` | prefetch 嘗試 | 企業 proxy 阻擋 GCS Range 請求 |
| 3 | 06-15 | ffmpeg remux 0610 會議音訊為正確 M4A | 單一會議可播放 | 新上傳仍是 raw AAC |
| 4 | 06-16 | Backend `/audio-stream` proxy 繞過企業防火牆 | CORS + proxy 阻擋 | 底層檔案仍無 moov atom |
| 5 | 06-16 | Frontend 改用 same-origin stream URL | 跨域問題 | raw AAC 無 duration metadata |
| **6** | **06-17** | **compose 後自動偵測 raw AAC → ffmpeg remux** | **✅ 根本修復** | — |

### 根本原因鏈
```
使用者上傳 .m4a（實為 raw ADTS AAC, 0xFFF1 開頭）
  → chunked upload 把原始 bytes compose 成 GCS blob
  → GCS blob 仍是 raw ADTS AAC（無 ftyp/moov/mdat）
  → backend proxy 正確串流 206 + audio/mp4
  → 瀏覽器 <audio> 讀到 raw AAC → 無法取得 duration
  → 播放器顯示 0:00/0:00
```

### 最終解決方案（Backend revision 00052）
在 `meeting_ops.py` chunked upload compose 完成後：
1. 下載 composed blob 到 tmpdir
2. 檢查前 2 bytes：若為 `0xFF 0xFx`（ADTS sync word）→ 需要 remux
3. `ffmpeg -y -i raw.aac -c copy -movflags +faststart remuxed.m4a`
4. 上傳 remuxed 檔案回 GCS（覆蓋原 blob）
5. 設定 `Content-Type: audio/mp4`

### 關鍵技術知識
- **ADTS AAC vs M4A**：ADTS 是 frame-by-frame 串流格式（每 frame 自帶 header），無全域 duration；M4A 是 MP4 container，moov atom 記錄完整 track info。
- **faststart**：將 moov atom 移到檔案開頭，使 HTTP streaming 可在下載完整檔案前就取得 duration。
- **企業環境**：奇美的 proxy 阻擋 browser 對外部 CDN（GCS）的 Range 請求，故音訊必須走 same-origin backend proxy。
