# DevLog 2026-06-18: Bulk Upload Stress Test & GPU Capacity Planning

## 背景

MeetChi 過去在使用者一次上傳多個大型會議檔案時，backend 會因 CPU 耗盡而被 LIVENESS probe 殺死（OOM-like crash）。本次實作 Cloud Tasks queue throttling 方案後，進行 5 場同時上傳壓力測試。

## 壓測設定

- 5 場會議，音檔長度 53-92 分鐘，總計 375 分鐘
- 音檔來源：UAT02 歷史資料（複製至獨立 meeting ID）
- 同時觸發：5 個 `POST /api/v1/tasks/enqueue-transcription` 在 1 秒內發出
- GPU 配置：maxScale=2, minScale=1, concurrency=5

## 發現的 Bug

### Bug 1: Cloud Tasks Retry 覆蓋已完成資料 (Critical)

**現象**：壓測-美玉 只有 18% 轉錄覆蓋率，但第一次執行實際成功。

**Root Cause**：
1. 第一次 dispatch 成功處理全部 6 chunks（05:09-05:29）
2. Cloud Tasks 因 HTTP response timeout 判定失敗 → 觸發 retry
3. 第二次執行先 `DELETE FROM transcript_segments WHERE meeting_id = ...`
4. 重新 ffmpeg split 後送 GPU，但 chunk_001-005 在 GCS 已被清理 → 404
5. 最終只存了 chunk_000 的 113 segments

**修復**（commit `eee1f5b`）：
```python
# 冪等性檢查：已 COMPLETED 的會議不再處理
if meeting.status == "COMPLETED":
    return {"status": "completed", "message": "Already completed, skipped"}
if meeting.status == "PROCESSING" and retry_count > 0:
    return {"status": "skipped", "message": "Already processing, skipped retry"}
```

### Bug 2: GPU maxScale=2 遠低於需求

壓測期間只有 2 個 GPU instance，5 場會議搶 10 個 slot（2×concurrency=5），導致大量排隊。

## GPU 容量規劃（產線情境）

### 需求估算

- 每日 0800-1700：100 場會議 × 2hr median = 200hr 音檔
- 黑數（大量上傳）：+50% = 100hr
- **總需求：~300hr 音檔 / 9hr 工作日**

### 單 GPU Instance 吞吐量

```
每 15min chunk → 7min GPU 處理（2.1x realtime）
每 instance concurrency=5
吞吐量 = 5 × (60÷7) ≈ 43 chunks/hr = 10.75hr 音檔/instance/hr
```

### 容量對照

| maxScale | 日容量 (9hr) | vs 需求 300hr | 尖峰餘裕 |
|----------|-------------|-------------|---------|
| 2 (舊) | 193 hr | ❌ 不夠 | 無 |
| 4 | 387 hr | ✅ 剛好 | 30% |
| **8 (新)** | **774 hr** | **✅ 充裕** | **158%** |

### 尖峰分析

- 10:00-11:00（晨會結束）：20-30 場同時觸發
- 所需：20 meetings × 8 chunks × 7min ÷ 30min ÷ 5 = **~7.5 instances**
- maxScale=8 可覆蓋

## 決策：GPU Infrastructure 調整

| 參數 | 舊值 | 新值 | 說明 |
|------|------|------|------|
| GPU maxScale | 2 | **8** | 覆蓋尖峰 + 黑數 |
| GPU minScale | 依排程 0/1 | **維持** | Mon-Fri 0800-1700 = 1, 其他 = 0 |
| Cloud Tasks maxConcurrentDispatches | 2 | **5** | 允許更多會議並行 |
| Backend ASR_PARALLELISM | 3 | **5** | 每場送更多 chunk |

### 排程維持不變

```
gpu-asr-schedule-up:   "0 8 * * 1-5" (Asia/Taipei) → minScale=1
gpu-asr-schedule-down: "0 17 * * 1-5" (Asia/Taipei) → minScale=0
```

## 壓測結論

1. **Cloud Tasks queue throttling 有效**：Backend 全程未崩潰 ✅
2. **GPU maxScale=2 是瓶頸**：需調整到 8 才能負載產線
3. **冪等性保護必要**：Cloud Tasks retry 會覆蓋正確結果（已修復）
4. **預估效能改善**：maxScale=8 + ASR_PARALLELISM=5 後，單場 2hr 會議可在 15-20min 完成
