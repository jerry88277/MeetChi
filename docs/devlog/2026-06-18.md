# DevLog 2026-06-18: Bulk Upload Stress Test & GPU Capacity Planning

## 背景

MeetChi 過去在使用者一次上傳多個大型會議檔案時，backend 會因 CPU 耗盡而被 LIVENESS probe 殺死（OOM-like crash）。本次實作 Cloud Tasks queue throttling 方案後，進行 5 場同時上傳壓力測試。

## 壓測設定

- 5 場會議，音檔長度 53-92 分鐘，總計 375 分鐘
- 音檔來源：UAT02 歷史資料（複製至獨立 meeting ID）
- 同時觸發：5 個 `POST /api/v1/tasks/enqueue-transcription` 在 1 秒內發出
- GPU 配置：maxScale=2, minScale=1, concurrency=5

## 發現的 Bug

### Bug 1: Cloud Tasks Retry 覆蓋已完成資料 (Critical)

**現象**：壓測-美玉 只有 18% 轉錄覆蓋率，但第一次執行實際成功。

**Root Cause**：
1. 第一次 dispatch 成功處理全部 6 chunks（05:09-05:29）
2. Cloud Tasks 因 HTTP response timeout 判定失敗 → 觸發 retry
3. 第二次執行先 `DELETE FROM transcript_segments WHERE meeting_id = ...`
4. 重新 ffmpeg split 後送 GPU，但 chunk_001-005 在 GCS 已被清理 → 404
5. 最終只存了 chunk_000 的 113 segments

**修復**（commit `eee1f5b`）：
```python
# 冪等性檢查：已 COMPLETED 的會議不再處理
if meeting.status == "COMPLETED":
    return {"status": "completed", "message": "Already completed, skipped"}
if meeting.status == "PROCESSING" and retry_count > 0:
    return {"status": "skipped", "message": "Already processing, skipped retry"}
```

### Bug 2: GPU maxScale=2 遠低於需求

壓測期間只有 2 個 GPU instance，5 場會議搶 10 個 slot（2×concurrency=5），導致大量排隊。

## GPU 容量規劃（產線情境）

### 需求估算

- 每日 0800-1700：100 場會議 × 2hr median = 200hr 音檔
- 黑數（大量上傳）：+50% = 100hr
- **總需求：~300hr 音檔 / 9hr 工作日**

### 單 GPU Instance 吞吐量

```
每 15min chunk → 7min GPU 處理（2.1x realtime）
每 instance concurrency=5
吞吐量 = 5 × (60÷7) ≈ 43 chunks/hr = 10.75hr 音檔/instance/hr
```

### 容量對照

| maxScale | 日容量 (9hr) | vs 需求 300hr | 尖峰餘裕 |
|----------|-------------|-------------|---------|
| 2 (舊) | 193 hr | ❌ 不夠 | 無 |
| 4 | 387 hr | ✅ 剛好 | 30% |
| **8 (新)** | **774 hr** | **✅ 充裕** | **158%** |

### 尖峰分析

- 10:00-11:00（晨會結束）：20-30 場同時觸發
- 所需：20 meetings × 8 chunks × 7min ÷ 30min ÷ 5 = **~7.5 instances**
- maxScale=8 可覆蓋

## 決策：GPU Infrastructure 調整

| 參數 | 舊值 | 新值 | 說明 |
|------|------|------|------|
| GPU maxScale | 2 | **8** | 覆蓋尖峰 + 黑數 |
| GPU minScale | 依排程 0/1 | **維持** | Mon-Fri 0800-1700 = 1, 其他 = 0 |
| Cloud Tasks maxConcurrentDispatches | 2 | **5** | 允許更多會議並行 |
| Backend ASR_PARALLELISM | 3 | **5** | 每場送更多 chunk |

### 排程維持不變

```
gpu-asr-schedule-up:   "0 8 * * 1-5" (Asia/Taipei) → minScale=1
gpu-asr-schedule-down: "0 17 * * 1-5" (Asia/Taipei) → minScale=0
```

## 壓測結論

1. **Cloud Tasks queue throttling 有效**：Backend 全程未崩潰 ✅
2. **GPU maxScale=2 是瓶頸**：需調整到 8 才能負載產線
3. **冪等性保護必要**：Cloud Tasks retry 會覆蓋正確結果（已修復）
4. **預估效能改善**：maxScale=8 + ASR_PARALLELISM=5 後，單場 2hr 會議可在 15-20min 完成

## 壓測5: 4.4hr 長影片壓力測試 (1.7GB MP4)

### 測試配置

- **影片來源**：YouTube `r-OR9QAqHRE`「2021精準醫學學術研討會」
- **規格**：1080p MP4, 1.7GB, 15935 秒 (4.4hr)
- **Meeting ID**：`9a69a4f4-07c9-4fa7-a868-54b13e3de50e`
- **GPU 配置**：maxScale=8, concurrency=8, ASR_PARALLELISM=14
- **觸發時間**：09:29:37 (UTC+8: 17:29)

### 時間線

| 時間 (UTC) | 事件 |
|---|---|
| 09:29:37 | enqueue (只產生 1 task ✅ atomic fix) |
| 09:30:07-43 | 18 chunks 分割完成並上傳 GCS (~35s) |
| 09:30:43 | 14 chunks 同時送出 GPU (parallelism=14) |
| 09:35:00-09:43:36 | 11/18 chunks 完成 |
| 09:40:27 | chunks 6,11,14,16,17 遭遇 ReadError → retry |
| 09:43:36 | chunk 16 retry 成功 (共 11/18 done) |
| 09:49:27 | GPU 完成 chunk_013, 但 backend 似乎 lost connection |
| 09:55+ | Backend asyncio.gather hanging (等待已斷開的連線) |

### 結果分析

1. **✅ Atomic idempotency 生效**：只建立 1 個 Cloud Task（前次 bug 會建 2 個）
2. **✅ Race condition fix 驗證**：第二次 dispatch 被正確攔截 (09:40:08 log)
3. **✅ 音檔分割正常**：4.4hr → 18 × 15min chunks，35秒完成
4. **✅ GPU 並行運作**：parallelism=14 正確，多 GPU instance 同時處理
5. **⚠️ ReadError 問題**：5/18 chunks 在 attempt 1 遭遇 ReadError
6. **❌ 長時間連線不穩定**：retry 後部分 chunks 仍 hang（httpx 連線斷開但未 timeout）

### 根因分析：ReadError

- GPU concurrency=8 配合 14 個並行請求 → 部分被 queue 在 instance 內
- Cloud Run GFE (Google Frontend) 的 TCP idle timeout (~15min) 會中斷長等待連線
- 當 GPU 處理完回傳時，原始 TCP 連線已被 GFE 斷開 → backend 收到 ReadError

### 後續改進方向

1. **降低 GPU concurrency**（8→2-3）：強制更多 instance，減少單 instance 排隊時間
2. **增加 retry 次數**（1→3）：ReadError 是暫時性錯誤，多 retry 可自癒
3. **考慮 exponential backoff**：retry 間隔從 2s 改為 2s/10s/30s

## Bug 修復總結

### Bug 3: Race Condition in Enqueue (Critical)

**現象**：同一 meeting 產生 2 個 Cloud Task（壓測5 初期發現）

**Root Cause**：
1. 兩個 enqueue 請求在 325ms 內到達（前端 double-request 或 retry）
2. 兩者都讀到 status=PENDING → 各自建立 task
3. 原本的 check 是 SELECT-then-UPDATE，非 atomic

**修復**（commit `47debba`）：
```python
# Atomic compare-and-swap: only claim if not already queued
result = db.execute(text("""
    UPDATE meetings
    SET processing_stage = 'queued', updated_at = NOW()
    WHERE id = :mid
      AND status::text NOT IN ('PROCESSING', 'COMPLETED', 'TRANSCRIBED')
      AND (processing_stage IS NULL OR processing_stage != 'queued')
"""), {"mid": request.meeting_id})
if result.rowcount == 0:
    return EnqueueResponse(status="skipped", ...)
```

### Bug 4: Transcription Endpoint Race Condition

**現象**：兩個 task 同時 dispatch，都通過 idempotency check

**修復**（同 commit `47debba`）：
```python
# Atomic claim: only one dispatch wins
result = db.execute(text("""
    UPDATE meetings
    SET status = 'PROCESSING', processing_stage = 'transcribing'
    WHERE id = :mid AND status::text IN ('PENDING', 'FAILED')
"""), {"mid": request.meeting_id})
if result.rowcount == 0:
    return {"status": "skipped", "message": "Claimed by another dispatch"}
```
