## 2026-04-13 - 資源/成本把關模式、進階錄音前處理、全域非同步佇列

### 🎯 動機（第一性原理）

1. **資源管制與成本把控**：當使用者同時上傳數個大型音檔、或是頻繁點擊重生摘要，將引發 Cloud Tasks 湧入過多請求，導致 Serverless 容器被水平擴展出數十個 Instance 並無限拉高 GCP 帳單（以及消耗 GPU Quota）。我們需要防重疊的邏輯，與強制最大的 Container Instances 來擔任物理限流的守門員。
2. **翻譯準確度與效能優化 (前處理)**：使用者的專有名稱（Custom Context）必須在模型開始運轉前注入 API 才能達到最好的翻譯/轉錄效果。音檔時長的解析若在後端運算需增加 I/O 以及轉碼開銷，我們應該將這兩項前置動作移轉到客戶端。
3. **無焦慮的多工 UX (全域佇列)**：當會議從「實體畫面阻塞」改為「背景處理」時，使用者在切換 Dashboard 其它頁面（如 RAG Workspace、Settings 等) 往往會失去對背景處裡的狀態感知，所以需要一個分離的前台指示器。

### 📦 變更檔案 MECE 清單

| 層級 | 檔案 | 變更 |
|------|------|------|
| **成本守門** | `backend-deploy.sh` | 增加 `--max-instances=1`，限制基礎設施的惡意/誤用擴展。 |
| **防覆蓋** | `main.py` | `regenerate_summary` 新增狀態判斷卡榫 `if meeting.status == "processing": ...` 避免任務重疊送出。 |
| **非阻塞 UX** | `api.ts` | 移除 `await` 並改用 `@catch` Fire-and-forget，避免 API 慢速回應卡死 UI 執行緒。 |
| **音檔預處理** | `useRecording.ts` | 利用原生的 `HTMLAudioElement` 與 `onloadedmetadata` 在上傳前擷取時長 `duration`。 |
| **Context**| `RecordingView.tsx`, `api.ts` | 新增「專有名詞 / 背景資料」輸入框（`customContext`），在處理鏈中賦予 LLM 專屬 Prompt 提示詞，並打通 WebSocket 與 regenerate API 的傳遞通道。 |
| **全域指標 UI**| `page.tsx` | 根據 `uploadState` 和 `isRegenerating` 設計全域佇列狀態標示（Processing queue pill），實現不阻塞的 Dashboard 切換。 |

### 💡 Wisdom

- **基礎設施與程式邏輯應雙管齊下**：軟體層的狀態防重疊只能防止介面端的問題，但若遭受惡意攻擊或程式 Bug 導致無窮迴圈的 API 轟炸，`--max-instances=1` 的硬體級限流就是確保 GCP 帳單不爆炸的最後防線。
- **發揮 Browser 的潛能**：善用客戶端原生的 `HTMLAudioElement` 功能，能夠避免後端動用 CPU 和套件（如 `ffmpeg`）處理純屬前置 Metadata 型態的工作，節省可觀的 I/O 與運算資源。

## 2026-04-13 - [UX 本質] Audio-Text 雙向同步連動完成 (Playback Sync)

### 🎯 動機（第一性原理）

轉錄的逐字稿與 RAG 檢索的關鍵字如果脫離原始的聲學脈絡，會產生強烈的「孤立感」。要達到最好的會議回顧體驗，使用者點擊文字或引用時，不僅能看，還必須能聽。我們需要透過 Player 的精準 Auto-Seek (跳轉)，把文字錨點和原始語境時間軸鎖定在一起。這是 Meeting Intelligence 工具的重要 UX 體驗底線。

### 📦 變更檔案 MECE 清單

| 層級 | 檔案 | 變更 |
|------|------|------|
| Frontend | `DetailView.tsx` | - 導入 `useRef<HTMLAudioElement>` 攔截播放器狀態<br>- 實作 `handleTimestampClick` 將時間字串轉為 float 的秒數並觸發 `.currentTime = sec; .play()`<br>- 監聽 `searchParams` (`?t=XX`) 在載入完畢且參數存在時自動 skip 到對應秒數播放 |
| Frontend | `RagService.ts` | Backend RAG API 改回傳具有 `start_time` 的 `Citation` 物件，修改介面匹配使其對齊 `start_time` |
| Frontend | `ChatPanel.tsx` | 配合 RAG API 修改，將 `response.references` 更新為 `response.citations` |
| Frontend | `ReferencePanel.tsx` | 將 Right-panel 引號連結 `<Link href="/dashboard/<id>">` 原生附加 `?t={reference.start_time}` URL 參數 |

### 💡 Wisdom

- **Data Flow vs UI Flow**：把時間狀態提升到 URL 參數 (`?t=xxx`) 是極佳的做法。如此一來不僅實現了 RAG Workspace 與 Dashboard 之間 Deep Link 的跳轉，使用者把該連結分享給他人時，其他人打開也能直接從討論焦點開始聽。
- **Browser Playback Restrictions**：現代瀏覽器有嚴格的 Autoplay policy，使用 `audioRef.current.play().catch(...)` 防禦是必要的，讓體驗即使遇到瀏覽器封鎖自動播放，也不會導致程式崩潰。

## 2026-04-08 - Phase 10.2 Fused MLA 原生 Cache 架構適配與 Windows 效能除錯

### 🎯 動機（第一性原理）

1. **推論階段相容性問題**：先前開發的 `WhisperMLAAttention` 自行維護 Cache Tensor，這與 Hugging Face `transformers` 原生的 `DynamicCache` 機制衝突。在呼叫 `model.generate()` 時，原生的機制會將 Cache 傳入 Tuple (而非 List)，導致 `type(past_key_value) is tuple` 判斷失敗。
2. **Windows 記憶體溢出（WinError 1455）**：當使用 `datasets` 處理大量語音資料準備 fine-tune 時，預設的 `num_proc=4` 在 Windows 作業系統上會瞬間拉起多個包含 PyTorch 和 GPU 驅動的完整子行程，快速耗盡分頁檔 (Page File)，導致程式全面崩潰。
3. **網路環境因素**：Hugging Face Hub 的資料集下載與驗證容易因為網路浮動或防火牆阻擋發生 HTTP 404 / 307，導致模型訓練前置作業無限 Timeout。

### 📦 修改內容

| 檔案 | 修改摘要 |
|------|---------|
| `whisper_mla/fused_mla.py` | [NEW] 實作 `WhisperMLAFusedAttention` 與 `WhisperMLAFusedModel`。我們不重新發明輪子，直接覆寫 `forward` 簽章，順著 HF `DynamicCache` 的 Tuple 結構（將 KV 封裝進 Tuple[0] 與 [1]）將 `past_key_value` 解析再組合，無縫對接 `model.generate()`。 |
| `whisper_mla/test_fused_inference.py` | [NEW] 建立推論對齊測試，確認我們修改過的 Cache 機制不僅能運作，輸出的 logits 與未修改的版本 cosine similarity 達到完全一致。 |
| `whisper_mla/train.py` | 針對 `WinError 1455`，明確在 `combined.filter(has_valid_text, num_proc=1)` 及 `combined.map(prepare_fn, num_proc=1)` 關閉 multiprocessing，妥協效能換取 Windows 下的系統穩定。原本的 Python import 從「腳本模式」改為 `-m whisper_mla.train` 套件模式解決 `ImportError: attempted relative import with no known parent package`。 |
| `PowerShell` | 整合離線訓練腳本，設定 `$env:HF_DATASETS_OFFLINE=1; $env:TRANSFORMERS_OFFLINE=1; $env:WANDB_MODE="offline"`，完全隔離網路波動，實現實質上的本地「離線訓練」。 |

### 💡 Wisdom
- **不要跟生態系對抗**：既然決定使用 `transformers` 的 `.generate()`，就必須透過 Wrapper (`FusedAttention`) 符合其 `DynamicCache` Tuple 結構，而不是去硬改 `DynamicCache` 的內核程式碼，這能大幅減少未來 HF 升級造成的技術債。
- **Windows 的多進程地雷**：遇到 `WinError 1455` (分頁檔太小) 時，增加虛擬記憶體是治標，治本之道是認清 PyTorch 在 Windows spawn 子行程的巨大 Overhead，改用單執行緒 `num_proc=1` 防範記憶體枯竭。
- **Python Module Context**：在包含相對導入的自定義資料夾架構中，必須嚴格遵守 `-m {package}.{module}` 啟動，否則層級將丟失而發生相對匯入崩潰。

## 2026-03-18~19 - Phase 10: Whisper-MLA ASR 引擎升級啟動

## 2026-03-12 - Phase 9 前端即時更新 + 通知系統架構諮詢

### 🔍 前端即時更新問題分析

**核心痛點**：`useMeetings.ts` 僅在 mount 時 fetch 一次，無任何 polling / WebSocket / SSE 機制。上傳錄音後 UI 卡在「上傳中」，需手動 F5 更新。

#### 五個事件觸發點（MECE 分析）

| 觸發點 | 事件 | 現有行為 | 應有行為 |
|--------|------|---------|---------|
| T1 上傳完成 | `uploadFile()` → API 201 | ✅ 正確 `fetchMeetings()` | — |
| T2 ASR 啟動 | `startTranscriptionTask()` → 202 | ❌ 單次 `fetchMeetings()` | 啟動 polling |
| T3 ASR 完成 | callback 更新 DB | ❌ 前端無感知 | polling 偵測 |
| T4 摘要完成 | `generate_summary_core()` → COMPLETED | ❌ 前端無感知 | polling 偵測 + toast |
| T5 摘要失敗 | exception → FAILED | ❌ 前端無感知 | polling 偵測 + error toast |

#### 架構決策（第一性原理）

| 決策 | 理由 |
|------|------|
| Event-Triggered Polling（非 timer-based） | Cloud Run Serverless 環境 SSE/WebSocket 有 timeout 限制，Polling 最穩定 |
| Exponential Backoff（5s→10s→15s, cap=30s） | 平衡即時性與 API 成本 |
| Page Visibility API | 背景分頁暫停 polling，節省資源 |
| 不使用 WebSocket | `useWebSocket.ts` 現有 hook 僅用於錄音串流，擴展為通用 status channel 會增加複雜度 |

### 🔔 Discord Webhook 通知 MVP

#### 通知 vs Polling 互補架構

```
Task COMPLETED
  ├→ [前端] Polling 偵測 status 變化 → UI 更新
  └→ [後端] Discord Webhook POST → 外部通知
```

- Polling = 面向**使用者**（UI 即時更新）
- Webhook = 面向**團隊**（跨平台通知）
- 兩者共享 `tasks.py` 中的 status 變更觸發點但各自獨立

通知訊息設計為 Discord Embed，包含完整時間線：
- `uploaded_at` → `asr_started_at` → `asr_completed_at` → `summary_completed_at`
- 會議標題、說話者數量、摘要模板類型

### 🔬 TranscriptHub 開源專案原始碼分析

深入讀取 7 個核心檔案（`main.js`、`task_controller.js`、`task_service.js`、`utils.js`、`router.go`、`server.go`、`index.html`），確認：

| 項目 | TranscriptHub | MeetChi |
|------|--------------|---------|
| 通知 | HTTPS POST → 自建 AI Portal（**非 Discord/Teams**） | Discord Webhook → Teams |
| 前端更新 | Go SSR + 記憶體快取 + **手動 F5** | React SPA + Event-Triggered Polling |
| Task Scheduler | `setInterval(10s)` DB polling | Cloud Tasks Queue |
| 可借鑑 | Fire-and-forget 模式、通知含下載 URL | ✅ |
| 不適用 | SSR/記憶體快取/無瀏覽器 polling | ❌ 架構差異過大 |

> **結論**：MeetChi 原定方案（Event-Triggered Polling + Discord Webhook）經 TranscriptHub 對照驗證，仍為最佳選擇。TranscriptHub README 宣稱「即時進度追蹤」實為 Go Server 記憶體快取 + 手動刷新。

- **痛點解決 (Root Cause Analysis)**：
  - **現象**：呼叫 Transcription API 經常卡死或 60 秒 Timeout 報 503 錯誤。
  - **根因**：經過架構檢視，確認 GCS FUSE 並不支援 SQLite 所需的 Byte-range File Lock。在 Cloud Run Serverless 多容器併發的情境下，SQLite 會引發永久性的 File Lock 死鎖。
  - **決策**：推翻先前退回 SQLite 的決定，重啟 Terraform 中的 `google_sql_database_instance` (PostgreSQL)。PostgreSQL 的 Client-Server 架構完美適配 Cloud Run，徹底決解 503 問題。
- **未來視角 (pgvector)**：PostgreSQL 原生支援 `pgvector`，這讓我們可以在同一個 `models.py` (SQLAlchemy) 中直接存放向量。未來的「語意搜尋(RAG)」不必再引入額外的向量資料庫 (如 Pinecone)，降低架構複雜度。

#### 2. 高機密調閱與防外流架構 (Traceability Focus)
- **第一性原理防禦盲點**：Web 前端不可能達到完美的「防匯出」 (因為有 Analog Hole: 手機實體翻拍螢幕 / 音訊外錄，以及 Network 攔截)，所以我們將防護策略設定為「Traceability (究責) > Absolute Prevention (絕對預防)」。
- **靜態資料庫防禦 (MECE - Data At Rest)**：拒絕在 FastAPI 中寫判斷式，改用 PostgreSQL **Row-Level Security (RLS)** 進行時限過濾。這樣即使擁有憑證的後端工程師直接連線 DB，沒拿到主管批准時間區間內的權杖，下達 `SELECT *` 也撈不出資料。
  - **⚠️架構毒藥警告**：由於 Cloud Run FastAPI 多半使用 Connection Pooling (`database.py` 中可能引入的 QueuePool)，實作 RLS 時必須依賴注入 `SET LOCAL application.user_id` 來隔離跨請求的權限污染。
- **動態畫面防護 (MECE - Data In Use)**：前端拋棄實體 `audio_url` 的讀取，改為後端動態簽發 Signed URL 阻絕直接下檔。畫面實作最高對比度的 SVG 動態浮水印，並結合 `MutationObserver` 避免技術人員透過 F12 刪除 DOM 節點。

#### 3. 機密等級設定與降級自動稽核 (Database Trigger + Outbox)
-   **防死角設計 (Anti-tamper)**：將「篡改、降級」的監控直接實作在 PostgreSQL 的 `AFTER UPDATE` Trigger 中，取代原先在應用層撰寫邏輯的想法。此舉確保即使是內部人員進行手動 SQL 操作也無法隱形。
-   **通報的一致性 (Outbox Pattern)**：將「稽核軌跡 (Audit Table)」的寫入與「通報工作清單 (Notification Queue)」的生成包被在同一個資料庫 Transaction 中。這完美解決分散式系統常見的 Dual-Write Problem（信寄了但 DB 沒換，或 DB 換了但斷線沒寄信）。Cloud Scheduler 在背景無腦重試拉取 Outbox 發信即可。

## 2026-03-10 - Callback 摘要觸發修復 + Cloud Tasks 入隊成功

### 🐛 根因（第一性原理）

`callbacks.py` 兩個邏輯缺陷：

| # | 問題 | 修復 |
|---|------|------|

### 📦 部署

| 服務 | 版本 | 變更 |

### ⚠️ 已知的 Race Condition（下次修復）

httpx 同步請求（1210s 後 TCP disconnect）早於 GPU ASR callback（17 秒差）。`generate_summary_core` 設 status=FAILED，但 callback 後來成功補位。E2E 腳本報 FAILED 但最終結果是 COMPLETED。

根因：GPU ASR 先 POST callback，再回覆 httpx HTTP response，但 httpx 連線已斷。

## 2026-03-12 - Phase 8 智能摘要引擎架構規劃

### 🏛️ 架構諮詢 × 4

以第一性原理 + MECE 原則，透過 NLM Research（18 學術來源 + 6 次 Query）完成 4 份架構諮詢報告並統合為 Phase 8 開發計畫：

| # | 報告 | 關鍵決策 |
|---|------|---------|
| 1 | 客製化摘要模板 | Schema-Driven Template Engine（JSON 配置 → 動態 Pydantic Schema → Gemini JSON Mode），取代硬編碼 `SummaryTemplate` |
| 2 | Role-Aware 摘要 | 方案 B (CoT 整合)：單次 Gemini 呼叫中注入「先分析角色再摘要」指令，零額外 API 成本 |
| 3 | 說話者標籤編輯 | Speaker Mapping Bar（inline 編輯 + 8 輔助色）+ `speaker_mappings` JSON 欄位（不新增表） |
| 4 | 優化 + 安全 | Meta-Prompting 優化按鈕（使用者預覽 diff 後決定）+ 三層 OWASP 防護（正則 → LLM 分類 → Sandwich Defense）|

### 📊 Phase 8 三階段拆解

- **8.1 基礎層**（Short）：Sandwich Defense + CoT 角色推斷 + Speaker Mapping Bar + Injection Guard
- **8.2 模板層**（Medium）：Template Engine + Gallery + Editor（含 10 系統預設模板）
- **8.3 增強層**（Medium-Large）：Meta-Prompting 優化 + AI 模板推薦 + 跨會議講者記憶 + Canary Token + 多視角摘要

### 💡 第一性原理決策

1. **CoT vs 兩階段呼叫**：選擇 CoT（零成本）而非獨立角色辨識 API（成本 ×2、延遲 ×2）
2. **JSON 欄位 vs 獨立表**：Speaker Mapping 與 Meeting 為 1:1，JSON 欄位最精簡
3. **Rewrite Pattern vs 自動替換**：使用者決定是否採用 AI 優化（Google PAIR 研究支撐）
4. **OWASP 發現**：`llm_utils.py:176` 將 `extra_instructions` 無消毒直接注入 prompt → P0 漏洞
