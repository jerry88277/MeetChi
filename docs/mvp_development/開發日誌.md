## 2026-03-04 - 企業資安與基礎設施架構諮詢完成

### 🎯 架構設計與決策記錄 (Architecture Consultation)

**狀態**: ✅ **諮詢完畢，排入開發任務**

透過深度勾稽 Terraform 現狀 (`database.tf` 中曾移除 Cloud SQL 改用 GCS FUSE)、後端架構 (`models.py` / `database.py`) 以及本日前半段處理的「Cloud Run 503 Service Unavailable」對話歷史，以 **MECE (相互獨立、完全窮盡)** 原則對系統進行了全方位的體檢與架構翻新規劃：

#### 1. 資料庫層級遷移 (SQLite -> Cloud SQL PostgreSQL)
- **痛點解決 (Root Cause Analysis)**：
  - **現象**：呼叫 Transcription API 經常卡死或 60 秒 Timeout 報 503 錯誤。
  - **根因**：經過架構檢視，確認 GCS FUSE 並不支援 SQLite 所需的 Byte-range File Lock。在 Cloud Run Serverless 多容器併發的情境下，SQLite 會引發永久性的 File Lock 死鎖。
  - **決策**：推翻先前退回 SQLite 的決定，重啟 Terraform 中的 `google_sql_database_instance` (PostgreSQL)。PostgreSQL 的 Client-Server 架構完美適配 Cloud Run，徹底決解 503 問題。
- **未來視角 (pgvector)**：PostgreSQL 原生支援 `pgvector`，這讓我們可以在同一個 `models.py` (SQLAlchemy) 中直接存放向量。未來的「語意搜尋(RAG)」不必再引入額外的向量資料庫 (如 Pinecone)，降低架構複雜度。

#### 2. 高機密調閱與防外流架構 (Traceability Focus)
- **第一性原理防禦盲點**：Web 前端不可能達到完美的「防匯出」 (因為有 Analog Hole: 手機實體翻拍螢幕 / 音訊外錄，以及 Network 攔截)，所以我們將防護策略設定為「Traceability (究責) > Absolute Prevention (絕對預防)」。
- **靜態資料庫防禦 (MECE - Data At Rest)**：拒絕在 FastAPI 中寫判斷式，改用 PostgreSQL **Row-Level Security (RLS)** 進行時限過濾。這樣即使擁有憑證的後端工程師直接連線 DB，沒拿到主管批准時間區間內的權杖，下達 `SELECT *` 也撈不出資料。
  - **⚠️架構毒藥警告**：由於 Cloud Run FastAPI 多半使用 Connection Pooling (`database.py` 中可能引入的 QueuePool)，實作 RLS 時必須依賴注入 `SET LOCAL application.user_id` 來隔離跨請求的權限污染。
- **動態畫面防護 (MECE - Data In Use)**：前端拋棄實體 `audio_url` 的讀取，改為後端動態簽發 Signed URL 阻絕直接下檔。畫面實作最高對比度的 SVG 動態浮水印，並結合 `MutationObserver` 避免技術人員透過 F12 刪除 DOM 節點。

#### 3. 機密等級設定與降級自動稽核 (Database Trigger + Outbox)
-   **防死角設計 (Anti-tamper)**：將「篡改、降級」的監控直接實作在 PostgreSQL 的 `AFTER UPDATE` Trigger 中，取代原先在應用層撰寫邏輯的想法。此舉確保即使是內部人員進行手動 SQL 操作也無法隱形。
-   **通報的一致性 (Outbox Pattern)**：將「稽核軌跡 (Audit Table)」的寫入與「通報工作清單 (Notification Queue)」的生成包被在同一個資料庫 Transaction 中。這完美解決分散式系統常見的 Dual-Write Problem（信寄了但 DB 沒換，或 DB 換了但斷線沒寄信）。Cloud Scheduler 在背景無腦重試拉取 Outbox 發信即可。

#### 根因分析（第一性原理 + MECE）

三個依賴版本 breaking change 同時發生：

| # | 根因 | 錯誤訊息 | 修復 |
|---|------|---------|------|
| 1 | `torchaudio>=2.9.0` 移除 `AudioMetaData` | `module 'torchaudio' has no attribute 'AudioMetaData'` | pin `torchaudio<2.6.0` |
| 2 | `huggingface_hub>=1.0` 移除 `use_auth_token` | `hf_hub_download() got unexpected keyword argument 'use_auth_token'` | pin `huggingface_hub<1.0` |

#### 交叉驗證來源
- pyannote-audio GitHub Issue #1804
- HuggingFace Discuss #172402/4
- PyTorch 2.6 Release Notes

#### 最終 `requirements-gpu.txt` 版本 pin
```
torch>=2.1.0,<2.6.0
torchaudio>=2.1.0,<2.6.0
huggingface_hub>=0.20.0,<1.0
```

#### Lessons Learned
> 1. ML pipeline 的每一個依賴都需要 semver 上限 pin — 只設下限是定時炸彈。
> 2. PyTorch / HuggingFace 的 major/minor 版本升級經常包含 breaking API changes。
> 3. 修復順序很重要：先固定版本避開 breaking change（穩定），再考慮升級到新 API（進步）。
> 4. `add_safe_globals` 白名單修復 PyTorch 2.6 weights_only 問題不可靠（型態數量不確定），pin 版本才是正解。

### 🔌 Phase 6.3: Webhook 解耦架構實作 (GPU ASR 回呼)

**狀態**: 🟢 **Completed**

針對自動摘要於 Cloud Run CPU 限流而導致中斷的問題，捨棄昂貴的 CPU Always Allocated 方案，改採 Cloud-Native 的 Webhook 解耦架構。

#### 完成項目
1. **GPU ASR 服務重構**：
   - 將原本阻塞在主執行緒的 ASR 運算邏輯，改為透過 FastAPI `BackgroundTasks` 在背景執行。
   - 主端點 `POST /asr/refine` 收單後立即回傳 `status: "processing"` (HTTP 202)。
   - 背景運算完成後，透過 `httpx.post` 將辨識結果與 `segments` 資料打回至 `callback_url`。
2. **CPU Backend 回呼支援**：
   - 新增 `POST /api/v1/callbacks/asr-done` 端點，作為 GPU ASR 運算完畢的專用接收器。
   - 接收到完成資料後，將更新原有的逐字稿標記與片段，並**在處理回呼的當下立刻觸發 `generate_summary_core` 產生 Gemini 摘要**。
   - 呼叫 GPU 服務時，帶入環境變數 `BACKEND_PUBLIC_URL` 來組成 `callback_url`。
3. **Terraform 連線字串修復 (Race Condition)**：
   - 部署時曾遭遇 `password authentication failed` 導致 Backend 崩潰的問題。
   - **Root Cause 1 (特殊字元解析失敗)**：Terraform 生成的新密碼含有保留字元 (`@`)，未經 `urlencode()` 處理便交給 SQLAlchemy 解析連線字串。
   - **Root Cause 2 (Race Condition)**：`cloudrun.tf` 未依賴 `google_sql_user.default`，導致 Cloud Run 先拿到新密碼，但 Cloud SQL 還未更新密碼。
   - **Fix**：補上 `depends_on = [google_sql_user.default]` 並在 `cloudrun.tf` 使用 `urlencode(random_password.db_password.result)`。

#### 核心決策與 First Principles
- **為什麼不用 CPU Always Allocated?**：強迫設定且搭配 `min-instances=1` 會喪失 Serverless 特性，產生每年近 $1,000 USD 閒置成本。改採 **Webhook 非同步回呼 (Decoupled Microservices)**，達到了完美的「0 閒置費用」，且徹底消除 Cloud Run HTTP Timeout 阻礙，未來耗時極長的分析任務也能順暢執行。



### 🏛️ Phase 7: 架構盤點與功能擴充規劃 (Consult)

**狀態**: 📝 **分析報告已產出**

針對目前 MVP 產品的問題，完成了一次系統架構盤點與 MECE 分析：

1. **講者標籤擴充**：決定不改動原先 `speaker` 字串架構，改為引入 `Speaker Mapping Table`，不僅向下相容，也能銜接 AD 帳號指派，實作 API `PATCH /api/v1/meetings/{id}/speakers` 最為直接。
2. **錄音檔時長計算**：放棄在後端下載 50MB 音檔解析，改在**前端上傳前透過 Web Audio API 解析 `duration`** 並寫入 DB，零成本、低延遲。

## 2026-03-07 - Webhook 非同步回呼 + Cloud Tasks 摘要解耦

### 🎯 核心問題

Cloud Run CPU 節流 (Throttle) 導致 `/tasks/transcription` API 回傳 202 後，`BackgroundTasks` 中的 Gemini 摘要流程被凍結。

### 💡 解決方案（第一性原理）

將摘要流程從 `BackgroundTasks` 解耦至 **Cloud Tasks Queue**（`meetchi-summarization-queue`），確保摘要邏輯在獨立的 Cloud Run Instance 上執行，不受原始 HTTP 回應的 CPU 節流影響。

### 🔧 修復的三個 Bug

| Bug | 根因 | 修復 |
|-----|------|------|
| Cloud Tasks enqueue 失敗 | Service Account 缺少 `cloudtasks.enqueuer` IAM 角色 | `gcloud projects add-iam-policy-binding` + Terraform `cloudrun.tf` |
| `/tasks/summarization` 迴圈觸發 GPU ASR | `generate_summary_core` 看到 `audio_url` 存在就再次觸發 ASR | 新增 `skip_asr=True` 參數，summarization 端點跳過 ASR 階段 |
| Gemini API 返回 500 | `gemini-2.5-flash-lite` 在 Vertex AI `asia-southeast1` 不可用 | 改用 `gemini-2.0-flash` |

### 📁 修改檔案

- `apps/backend/app/routes/callbacks.py` — Cloud Tasks 取代 BackgroundTasks
- `apps/backend/app/routes/cloud_tasks.py` — `skip_asr=True`
- `apps/backend/app/tasks.py` — 新增 `skip_asr` 參數
- `terraform/cloudrun.tf` — IAM + Gemini model 環境變數

### ✅ E2E 驗證結果

- 上傳 48.5MB 音檔 → GPU ASR 轉錄 33 分鐘音檔（798 segments, 8 speakers） ✅

## 2026-03-12 - Phase 8 智能摘要引擎架構規劃

### 🏛️ 架構諮詢 × 4

以第一性原理 + MECE 原則，透過 NLM Research（18 學術來源 + 6 次 Query）完成 4 份架構諮詢報告並統合為 Phase 8 開發計畫：

| # | 報告 | 關鍵決策 |
|---|------|---------|
| 1 | 客製化摘要模板 | Schema-Driven Template Engine（JSON 配置 → 動態 Pydantic Schema → Gemini JSON Mode），取代硬編碼 `SummaryTemplate` |
| 2 | Role-Aware 摘要 | 方案 B (CoT 整合)：單次 Gemini 呼叫中注入「先分析角色再摘要」指令，零額外 API 成本 |
| 3 | 說話者標籤編輯 | Speaker Mapping Bar（inline 編輯 + 8 輔助色）+ `speaker_mappings` JSON 欄位（不新增表） |
| 4 | 優化 + 安全 | Meta-Prompting 優化按鈕（使用者預覽 diff 後決定）+ 三層 OWASP 防護（正則 → LLM 分類 → Sandwich Defense）|

### 📊 Phase 8 三階段拆解

- **8.1 基礎層**（Short）：Sandwich Defense + CoT 角色推斷 + Speaker Mapping Bar + Injection Guard
- **8.2 模板層**（Medium）：Template Engine + Gallery + Editor（含 10 系統預設模板）
- **8.3 增強層**（Medium-Large）：Meta-Prompting 優化 + AI 模板推薦 + 跨會議講者記憶 + Canary Token + 多視角摘要

### 💡 第一性原理決策

1. **CoT vs 兩階段呼叫**：選擇 CoT（零成本）而非獨立角色辨識 API（成本 ×2、延遲 ×2）
2. **JSON 欄位 vs 獨立表**：Speaker Mapping 與 Meeting 為 1:1，JSON 欄位最精簡
3. **Rewrite Pattern vs 自動替換**：使用者決定是否採用 AI 優化（Google PAIR 研究支撐）
4. **OWASP 發現**：`llm_utils.py:176` 將 `extra_instructions` 無消毒直接注入 prompt → P0 漏洞
