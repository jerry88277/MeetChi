# /update-docs - 開發文件同步與維護習慣（v2）

作為一個專業的 Agent，當你在專案中完成了任何重要的功能實作、Bug 修復、或是輸出架構諮詢報告後，**必須主動實行**此工作流，確保專案進度與知識正確傳承。

## 步驟 1：檢閱當前狀態
- 讀取 `d:\Side_project\MeetChi\docs\mvp_development\開發任務.md`
- 讀取 `d:\Side_project\MeetChi\docs\mvp_development\開發日誌.md`

## 步驟 2：維護《開發任務.md》
- **標記完成**：將剛剛完成的子任務或 P0/P1 項目打勾 `[x]`。
- **排入新發現**：若在實作或除錯過程中發現了新的技術債、盲點或需要後續補強的架構方案，主動將其分門別類排入對應的 Sprint 或 Phase 中。

## 步驟 3：交叉比對（防遺漏）

> **此步驟是防止同一工作項目出現在多個區塊時只更新其中一處的關鍵機制。**

- 對每個剛完成的項目，用 `grep_search` 搜尋其關鍵字（如 `Cloud SQL`、`Webhook`、`pgvector`、`Gemini` 等）。
- 確認同一項目在**其他區塊**（Sprint、Phase、P0/P1/P2）的 checkbox 也同步更新為 `[x]`。
- **格式一致性規則**：嚴格禁止 `[ ] **xxx**: ✅ 已完成` 這種矛盾格式。完成就用 `[x]`，未完成就用 `[ ]`，不可混搭。

## 步驟 4：更新元資料

- **`最後更新` 日期**：更新為當天日期。
- **`文件版本` 號**：遞增並更新描述（如 `3.3（v6.3 完成）`）。
- **Image 部署表格**：若有重新部署，更新對應服務的 Image Tag、部署時間、本地一致性狀態。

## 步驟 5：撰寫《開發日誌.md》
- **建立標籤**：在當日日期的區塊下，新增一個標題紀錄剛完成的工作。
- **成果摘要**：簡明扼要地列出已實作的功能、解決的核心痛點（Root Cause 分析）與重要決策（Trade-offs）。
- **保留脈絡**：寫下為何這樣做（第一性原理），而不只是做了什麼，確保未來的開發者（或未來的你）能看懂原始動機。

## 觸發時機 (Trigger)
1. 用戶明確下達 `/update-docs` 指令時。
2. 每次完成一個複雜的 Bug 修復或告一段落時（**Agent 應具備此慣性，在通知用戶完成任務前，主動向用戶提議或直接執行上述更新**）。
3. 執行完 `/consult` 或 `/planning` 產生了明確的架構決策時。
4. `/ultrawork` 的 Enforcer 階段（完成前強制執行）。
